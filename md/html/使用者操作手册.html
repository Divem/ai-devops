Agentic·DevOps 用户手册（使用者版）

定位：帮助使用者快速用起来，并在遇到问题时可自助定位与解决。

快速上手 模块价值 操作指南 FAQ

1. 手册目标

这份手册面向日常使用系统的用户（产品经理、研发负责人、项目经理、需求协同人员），帮助你快速理解系统并高效使用。

  1. 快速理解系统定位与核心价值
  2. 明确每个模块“有什么用、什么时候用、怎么用”
  3. 遇到问题时在 FAQ 中快速找到处理方法

2. 系统定位与核心价值

2.1 系统定位

Agentic·DevOps 是一个 AI 驱动的需求管理与协作工作台。它将需求从评审到文档产出再到执行协同放在同一条流程中，减少信息断层和重复沟通。

2.2 核心价值

  • 更快：AI 辅助评审和文档生成，减少从零开始写作
  • 更稳：关键环节有结构化校验和补全，降低遗漏风险
  • 更清晰：需求状态、评审结果、文档进展集中可见
  • 可追溯：澄清记录、对话上下文、文档内容可持续复用

3. 整体功能地图

系统主要由 5 个功能区组成：

  1. 需求评审看板：管理需求状态流转和优先级
  2. 需求详情 / AI 设计页：查看需求上下文、生成与编辑文档
  3. Chatbot 与参考区：围绕当前需求持续澄清、补充和追问
  4. 数据概览：查看需求规模、阶段分布和过程质量
  5. 项目配置：管理模型、模板与协作配置

建议路径：看板 -> 详情 -> AI 评审 -> 文档生成 -> 回到看板推进。

4. 模块说明：价值与用法

4.1 需求评审看板

模块价值：统一查看需求阶段，快速识别高优先级和阻塞项，为评审会和迭代会提供同一事实视图。

你可以做什么：按业务维度筛选需求；打开需求详情；推进需求状态。

推荐用法：每天先看“高优先级 + 待评审”；评审会前筛选当前迭代；优先处理长期停留需求。

4.2 AI 智能评审（在需求详情中触发）

模块价值：在评审前发现信息缺口，输出结构化评分和风险提示，提高讨论质量。

你可以做什么：发起 AI 评审；查看风险清单与优化建议；补齐需求描述、用户故事、验收标准。

推荐用法：对 P0/P1 需求先评审再开会；低分需求先补全再复评；关键风险转为任务跟踪。

4.3 需求详情 / AI 设计页（文档中心）

模块价值：在同一上下文中完成文档生成和编辑，减少信息断层，提升产出速度与规范性。

你可以做什么：生成 PRD / Proposal / Design / Spec / Tasks；编辑与预览；迭代多版文档。

推荐用法：先 PRD，再 Spec，最后 Tasks；优先人工检查边界条件、异常流程、验收标准。

4.4 Chatbot 与参考资料区

模块价值：围绕当前需求连续对话，不丢上下文，快速补齐遗漏点并复用历史澄清。

你可以做什么：追问边界/异常/依赖/验收；用结构化方式表达（支持 Markdown 预览）；查看澄清摘要与历史问答。

推荐提问：“补充遗漏验收标准”“拆分业务/技术风险”“拆解任务并标注优先级”。

4.5 数据概览

模块价值：管理者看趋势，执行者看瓶颈，用数据判断流程健康度并支撑复盘。

你可以做什么：查看需求总量、阶段分布和处理进度；识别积压阶段与高风险区间。

推荐用法：每周固定查看阶段变化；对积压阶段建立专项动作；结合通过率与文档覆盖率分析。

4.6 项目配置（模型与模板）

模块价值：统一团队模型和输出规范，提升文档生成质量稳定性。

你可以做什么：选择并管理 AI 模型；配置文档模板和技能内容；调整默认生成风格。

推荐用法：先统一默认模型与模板，再根据评审反馈迭代，保留“标准模板 + 团队补充规则”。

5. 新用户快速上手路径

  1. 在看板选一条需求，阅读核心字段
  2. 触发 AI 评审，理解评分和风险建议
  3. 进入 AI 设计页生成 PRD 和 Tasks
  4. 在 Chatbot 追问缺失点并补全文档
  5. 回到看板推进状态并观察数据概览变化

6. 常见问题 FAQ

Q1：为什么触发 AI 评审后会先出现补充问题？

这是完整性校验机制。系统会先补齐关键信息，再发起评审，以提升评审准确性并减少无效结果。

Q2：AI 给了很多建议，我应该先改哪里？

优先顺序：用户故事是否明确 -> 验收标准是否可测试 -> 边界与异常是否覆盖。

Q3：文档很多，不知道先看哪份？

建议先 PRD（目标与范围），再 Spec（实现细节），最后 Tasks（执行拆解）。

Q4：Chatbot 怎么问，才能得到可执行答案？

用结构化问题：指定输出格式、指定维度、指定场景（正常/异常/边界）。

Q5：文档生成内容太泛怎么办？

先补充背景目标、关键约束、用户画像和验收标准，再基于同一会话追问修订。

Q6：同一需求反复迭代，怎么避免混乱？

建议每次按“先澄清变更点 -> 再更新 PRD/Spec -> 最后同步 Tasks”执行。

Q7：数据概览看哪些指标最有用？

优先关注各阶段需求数量、已通过占比、文档覆盖情况。

Q8：输出不符合预期时怎么办？

不要重开流程，先补充上下文，再明确输出要求，最后在同一会话中追问修订。

7. 实用技巧

  • 先筛选再评审，避免无关需求干扰
  • 先做结构检查，再做措辞润色
  • 沉淀常用提问模板为团队统一话术
  • 固定用数据概览做短周期复盘
