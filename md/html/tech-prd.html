版本：2026-03-10 ｜ 代码基线：pm-ai-app/src/PMPlatform.jsx ｜ 规格基线：openspec/specs/

看板流转 AI 评审 澄清问答 文档中心 Chatbot 稳定性

一、背景与目标

背景

需求评审、文档编写、提案确认和 Git 交付长期分散在多个工具，导致上下文反复搬运、信息不一致和反馈延迟。当前产品演进目标是把“高频标准动作”固化为稳定闭环，并确保关键路径可追溯、可恢复、可演示。

目标

  • 形成端到端链路：看板推进 -> AI 评审 -> 澄清补全 -> 文档生成 -> Git 提交。
  • 建立过程证据链：澄清摘要、消息状态、错误回退均可在会话中追踪。
  • 提升使用效率：顶部模型快速切换、三栏拖拽布局、文档树分组检索。

非目标

  • 本期不建设完整后端持久化和权限体系。
  • 本期不覆盖多人实时协同编辑和评论工作流。

二、角色与核心场景

角色核心任务关键诉求
产品经理评审需求、补充澄清、生成文档减少重复解释，提升输出质量
研发负责人判断可行性和风险快速看到结构化评审结果
研发工程师消费 Spec/Tasks 并执行交付内容完整且可追踪
售前/演示角色稳定演示端到端流程失败场景也能不断链

三、功能范围

In Scope

  • 看板：状态流转、筛选、新建/导入。
  • AI 评审：完整性检测、补充问卷、评审结果与兜底。
  • 文档中心：PRD + proposal/design/spec/tasks 生成、编辑、提交。
  • 澄清流程：生成前问答、Q/A 摘要入 Chatbot、摘要参与 prompt。
  • Chatbot：Markdown 预览、乐观更新、thinking trace、fallback 与重试。
  • 效率增强：顶栏模型切换、三栏拖拽持久化、文档树分组。

Out of Scope

  • Meego 全量双向同步。
  • 版本历史 + 批注 + 分屏对比完整协作栈。
  • 提案包级提交与 PR/MR 自动创建。

四、关键流程（技术实现口径）

流程 A：AI 评审

  1. 触发评审请求。
  2. 校验必填字段（title/desc/priority/userStory/acceptanceCriteria）。
  3. 若缺失则弹框补充；校验通过后发起评审。
  4. 结果写入 aiResult；失败时走 fallback 展示链路。

流程 B：PRD/SPEC 生成前澄清

  1. 点击生成文档。
  2. 根据语义缺口触发澄清问卷。
  3. 提交问答后构建 Q/A 摘要并写入 chatHistory_{model}。
  4. 将摘要拼接到生成 prompt，执行文档生成。

流程 C：Chatbot 发送链路

  1. 本地先插入用户消息与思考占位（pending）。
  2. 异步请求 AI，按消息 ID 原位替换为 done。
  3. 失败时替换为 fallback 并保留重试入口。

五、数据模型与状态约束

对象关键字段约束
需求卡片id/col/priority/title/desc/userStory/acceptanceCriteria/aiResult/docs评审与生成过程中状态一致性必须保证
文档对象docs.prd + docs.proposals[]proposal/design/spec/tasks 可部分成功，不阻断全链路
会话历史chatHistory_claude/glm/ark/custom按模型隔离，需支持澄清摘要与 fallback 消息
布局状态leftWidth/rightWidth需 clamp 最小/最大值并持久化恢复

六、验收标准

  • 至少 1 个模型跑通“评审 -> 澄清 -> 生成 -> 提交”链路。
  • 澄清问答可在 Chatbot 可视并可复用到生成上下文。
  • Chatbot Markdown 预览与最终渲染一致。
  • 失败场景有会话内 fallback 展示并可重试。
  • 三栏拖拽与重进恢复行为可验证。

七、后续迭代建议

  1. P0：提案包级 Git 提交，降低交付操作摩擦。
  2. P0：Meego 最小可用同步，打通源头与平台数据。
  3. P1：澄清摘要结构化回填（自动更新 userStory/acceptanceCriteria）。
  4. P1：版本快照与 diff，支撑多人协作评审。
