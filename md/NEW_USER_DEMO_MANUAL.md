# PM AI Platform 用户手册（新用户版）

> 适用对象：第一次使用 PM AI Platform 的产品经理、项目经理、研发负责人。

---

## 1. 这是什么平台？

PM AI Platform 是一个面向需求管理与方案产出的 AI 工作台。它把需求从“收集、评审、设计、文档化、复盘”放进同一条流程，避免需求信息散落在 IM、文档和表格里。

一句话：**用看板管理进度，用 AI 提升质量，用文档沉淀结果，用数据做复盘。**

---

## 2. 主要解决什么问题？

平台重点解决 4 类常见问题：

1. **需求信息不完整**：AI 评审会从完整性、逻辑性、风险三个维度给出结构化建议。
2. **评审效率低、主观性强**：统一流程和评分视图，降低“各说各话”。
3. **文档产出慢、质量不稳定**：AI 可快速生成 PRD / Proposal / Design / Spec / Tasks 初稿。
4. **项目过程不可复盘**：数据概览按维度筛选，看到团队当前瓶颈与产出情况。

---

## 3. 平台功能地图

### 3.1 需求评审看板

- 7 个阶段管理需求生命周期：待评审、评审中、AI 分析中、人工确认、已通过、已拒绝、已提交（git commit）。
- 支持拖拽流转、卡片快速浏览、多维筛选。
- 筛选维度：空间、子系统、应用、迭代。
- 新建需求入口位于筛选行右侧，便于边筛选边创建。

> 说明：`已提交（git commit）` 表示需求文档已提交到 Git，产品阶段阶段性收尾。

### 3.2 需求详情页（文档 + AI）

- 左侧文档树：支持 PRD 及多提案文件夹（Proposal / Design / Spec / Tasks）。
- 中间文档区：支持预览与编辑。
- 右侧助手区：支持 AI 对话与参考资料。
- 原始需求视图：在无真实数据时，会展示来自 `PM_AI_Plagform_PRD` 的示例摘录，并标注“示例内容（来自 PRD 摘录）”。

### 3.3 AI 智能评审

- 一键评审需求内容。
- 输出总分及三维评分（完整性、逻辑性、风险）。
- 自动给出风险点和修改建议，可直接用于完善需求。

### 3.4 AI 文档生成

- 可针对同一需求生成多类文档。
- 支持多提案并行方案沉淀。
- 支持人工二次编辑，形成可执行版本。

### 3.5 项目配置

- AI 模型配置：支持 Claude、GLM-4、ARK Code Latest。
- API Key 本地保存（浏览器 localStorage）。
- Git 仓库配置、SDD 框架模板配置、AI Skill 配置。

### 3.6 数据概览

- 展示需求总量、待处理、已通过、文档覆盖率、阶段分布。
- 支持按阶段/优先级/空间/迭代等维度下钻筛选。

## 4. 快速上手（10 分钟）

### 第一步：启动系统

在 `pm-ai-app` 目录执行：

```bash
cd pm-ai-app
npm install
npm run dev
```

访问：`http://localhost:5173`

### 第二步：配置 AI 模型（推荐）

右上角 ⚙ → 项目配置 → AI 模型配置：

- 选择模型（Claude / GLM-4 / ARK）
- 输入对应 API Key
- 点击“保存 AI 配置”

> 不配置也可使用：AI 请求失败时，系统会回退到示例数据，便于体验流程。

### 第三步：创建或选择需求

- 在看板筛选行右侧点击“新建需求”，或直接选择现有卡片。
- 补充最关键字段：标题、描述、用户故事、验收标准。
- 如原始需求暂缺，可先参考系统自动填充的 PRD 摘录示例，再补录真实信息。

### 第四步：执行 AI 评审

- 进入需求详情页，点击“AI 智能评审”。
- 查看分数和建议，按建议补充需求内容。

### 第五步：生成文档并进入执行

- 在详情页文档树中生成 PRD / Proposal / Design / Spec / Tasks。
- 完善后把 Tasks 交付研发排期。
- 文档确认后点击“提交到仓库”，成功后需求状态会进入“已提交（git commit）”。

## 5. 推荐使用流程（日常）

1. **需求录入**：新建需求并补全基础字段。
2. **AI 初评**：先跑一轮 AI 评审，找到缺口。
3. **人工确认**：评审会只讨论关键问题。
4. **文档生成**：生成并整理方案文档。
5. **任务拆解**：基于 Tasks 进入开发排期。
6. **复盘优化**：在数据概览中观察趋势并调整流程。

## 6. 常用操作指引

### 6.1 看板筛选

- 空间：按业务线或组织维度过滤。
- 子系统：按系统域过滤。
- 应用：按具体应用过滤。
- 迭代：按发布节奏过滤。

### 6.2 文档树使用

- PRD 通常作为顶层产品需求说明。
- 原始需求为只读展示区，优先显示真实 Meego/人工录入数据；无数据时显示 PRD 摘录示例。
- 多提案文件夹用于并行方案（A/B 方案）对比。
- Proposal、Design、Spec、Tasks 建议成组维护，保持一致性。

### 6.3 AI 对话使用

可以直接提问：

- “这条需求最容易遗漏哪些验收标准？”
- “请把风险分为业务风险和技术风险。”
- “把当前内容整理成可评审版本。”

## 7. 常见问题（FAQ）

### Q1：提示“未配置 API Key”

- 进入项目配置，确认模型与 Key 一致。
- Claude 用 Claude Key，GLM 用 GLM Key，ARK 用 ARK Key。

### Q2：模型切换后效果不明显

- 确认已点击保存配置。
- 刷新页面后重新发起 AI 请求。

### Q3：AI 请求失败还能继续吗？

- 可以，系统有演示回退数据，不影响流程体验。

### Q4：文档太多怎么管理？

- 按“一个需求 = 一个 PRD + 若干提案文件夹”管理。
- 每次评审只推进一个主方案，其他方案保留对照。

### Q5：为什么我看到“示例内容（来自 PRD 摘录）”？

- 这是演示兜底机制：当原始需求为空时，系统会自动展示示例摘录。
- 一旦你录入或同步真实原始需求，示例内容会自动消失。

## 8. 新用户首周建议

- Day 1：录入 5-10 条真实需求，统一字段模板。
- Day 2：对高优先级需求执行 AI 评审并人工复核。
- Day 3：为 2-3 条重点需求生成 Spec + Tasks 并排期。
- Day 4-5：沉淀一版团队评审标准与文档模板。
- 周末：通过数据概览做一次小复盘（通过率、滞留阶段、文档覆盖率）。
