# PM AI Platform 设计系统分析

## 1. 结论先行

该项目的设计系统已经形成了**可落地的基础 Token + 业务语义色 + 内联样式组件模式**，并在多个版本中保持了较高一致性。当前主应用（`pm-ai-app/src/PMPlatform.jsx`）相较历史预览版本（`pm-ai-platform_sdd.jsx`、`pm-ai-platform_sdd2.jsx`、`pm-ai-platform_vibe.jsx`）演进明显，核心特征是：

- 以 `C` 作为全局颜色 Token 源，覆盖中性色、功能色、侧边栏色。
- 以看板状态（`COLUMNS`）、优先级（`P0-P3`）、AI 分数（高/中/低）构建业务语义映射。
- 通过浅色/深色双套 token（主应用）与设计工作台子主题（历史版本）实现主题切换。
- 组件层采用“内联样式 + 函数式渲染”的单文件模式，迭代速度快，但复用与治理成本高。

整体定位不是通用 UI 库，而是**“产品需求工作台场景下的任务型设计系统”**。

---

## 2. 设计系统分层结构

从代码看，项目的设计系统可分为 4 层：

1. **基础视觉 Token 层**
   - 入口：`const C = { ... }`
   - 提供色板：`ink/paper/cream/white`、`accent/success/warn/danger/purple/teal`、`sb*`

2. **主题与语义映射层**
   - 主应用：`LIGHT_THEME_TOKENS` / `DARK_THEME_TOKENS` + `applyThemeModeTokens()`
   - 历史版本：`AI_DESIGN_THEME.dark/light`（聚焦设计工作台局部）

3. **业务状态视觉层**
   - 看板列状态：`COLUMNS`
   - 优先级映射：`priorityColor()`
   - 评分映射：`scoreColor()` + `scoreBg()`
   - 文档类型语义：`DOC_TYPES`

4. **页面组件实现层**
   - 卡片、抽屉、Tree、Tag、Button、Toast、Modal 等都用内联样式直接消费上述 Token

这套分层说明：项目已经具备“系统性”，但尚未抽离为独立的组件包和样式基础设施。

---

## 3. 视觉语言（Color / Typography / Shape）

## 3.1 颜色体系

### 中性色
- `ink`: 主文字深色
- `paper/cream/white`: 页面背景、容器底色、输入区底色分层
- `muted/border`: 次级文本与边框

### 功能色
- `accent`: 主操作与链接
- `success/warn/danger`: 状态反馈与风险提示
- `purple/teal`: AI/文档/辅助模块强调色

### 场景色
- `sb/sbHover/sbActive/sbText/sbMuted`: 侧边文档树深色主题专用色

**观察**：项目把“品牌色”与“业务状态色”分离，且大部分界面都能通过 Token 回收颜色，属于可维护的设计方向。

## 3.2 字体体系

- 主体阅读：`'Noto Sans SC', sans-serif`
- 编号/状态/技术信息：`'DM Mono', monospace`

这种“双字体策略”使得中文业务文本可读性与结构化信息扫描效率同时兼顾，适合需求管理、审阅、文档树等场景。

## 3.3 形状与层级

- 圆角以 `4/6/8/10/12` 为主，卡片和弹层统一偏圆角
- 阴影用于抽屉、弹窗、下拉层，层级关系清晰
- 边框普遍 1px，配合浅背景形成“轻分隔”视觉语义

---

## 4. 业务语义设计（这是该项目最有价值的部分）

## 4.1 看板状态语义

主应用中看板状态已经覆盖到提交阶段：

- `backlog` 待评审
- `reviewing` 评审中
- `ai_review` AI分析中
- `confirm` 人工确认
- `approved` 已通过
- `rejected` 已拒绝
- `submitted` 已提交（git commit）

每个状态绑定专属 `color/bg`，使用户在不看文案时也可通过色彩快速识别流程位置。

## 4.2 优先级语义

- `P0`: 红色（高风险/最高优先）
- `P1`: 橙色
- `P2`: 蓝色
- `P3`: 中性色

语义直观，符合多数工程/产品团队认知模型。

## 4.3 评审分数语义

- `>=80`: success
- `60-79`: warn
- `<60`: danger

通过 `scoreColor + scoreBg` 双通道传递强弱，不只依赖单一文字。

---

## 5. 主题系统与多版本关系

## 5.1 当前主应用（pm-ai-app）

主应用采用全局 token 主题切换：

- `readThemeMode()` 从 `localStorage` 读取模式
- `applyThemeModeTokens()` 直接覆写 `C`，并同步更新 `COLUMNS` 配色

特点：主题能影响主界面全局，不仅是局部区域。

## 5.2 历史版本（sdd / sdd2 / vibe）

历史版本采用 `AI_DESIGN_THEME.dark/light`，重点作用于设计工作台（顶部、文档树、编辑区），主看板区相对固定。

## 5.3 版本差异要点

- `sdd`: 文档类型含 `spec`
- `sdd2` / `vibe`: 文档类型偏 `prd/proposal/design/tasks`
- 主应用新增 `raw` 文档类型与 `submitted` 流程状态，语义更完整

说明项目演进方向是：从“演示型多变体 UI”走向“统一主应用 + 业务全链路状态系统”。

---

## 6. 组件与交互模式总结

从实现看，项目虽未拆分成大量独立组件文件，但已形成稳定交互模式：

- **看板卡片模式**：左侧状态色条 + 优先级角标 + 标签 + AI 分数
- **抽屉详情模式**：右侧固定宽度抽屉，承载评审、文档、原始需求
- **文档树模式**：深色导航 + 层级缩进 + Badge 计数 + 状态点
- **弹窗模式**：中大尺寸居中 Modal，统一边框、圆角、阴影
- **按钮模式**：主按钮（accent/purple）、次按钮（白底描边）、危险按钮（danger）
- **反馈模式**：Toast、状态标签、颜色型告警块

这些模式在不同页面重复出现，说明系统“隐式组件规范”已经存在。

## 7. 当前优势与约束

## 7.1 优势

- 业务语义与视觉语义绑定紧密（状态、优先级、评分）
- 主题与 Token 已有基础，具备继续平台化的可能
- 视觉风格统一，信息密度高，适合中后台/工作台场景

## 7.2 约束

- 样式以内联对象为主，复用依赖复制粘贴，统一改动成本高
- Token 仍偏“颜色主导”，缺少显式 spacing/typography/elevation token 字典
- 主应用为超大单文件，设计系统规则难被自动化约束

---

## 8. 建议的下一步（按优先级）

1. **提炼语义 Token**
   - 在 `C` 之上增加 `semantic` 层：`bg.surface/card/panel`、`text.primary/secondary`、`state.info/success/warn/danger`

2. **抽离基础样式原语**
   - 先提取 `Button/Tag/Badge/Card/Drawer/Modal` 五类高频原语，保持现有视觉不变，仅做复用收敛

3. **引入 CSS 变量桥接主题**
   - 保留 JS Token，同时将关键色落到 `--color-*`，降低主题切换的渲染和维护复杂度

4. **建立设计系统文档与示例页**
   - 在 `md/` 或预览子应用中增加“色板/状态/组件”单页，作为协作基线

## 9. UI 设计规范补充（可直接执行）

以下规范用于后续页面新增或改版时的统一约束，优先遵循“与现有视觉一致”的原则。

## 9.1 布局与栅格

- 页面外层保持工作台结构：`top nav + content`，高度基于 `100vh`。
- 主内容区默认 2~3 列：左侧导航/文档树（240~300px）+ 主工作区（自适应）+ 右侧抽屉（52~60vw，最小 520px）。
- 区块间距使用 8pt 体系：`4/8/12/16/24/32`，优先使用 `8/12/16`。
- 卡片内边距建议 `12~16`，弹窗内边距建议 `20~24`，确保信息密度与可读性平衡。

## 9.2 排版规范

- 字体族：正文使用 `Noto Sans SC`，编号/状态/技术元信息使用 `DM Mono`。
- 字号层级建议：
  - 页面标题：`20~26` / `600~800`
  - 区块标题：`14~16` / `600~700`
  - 正文：`13~14` / `400~500`
  - 辅助信息：`10~12` / `500~700`
- 行高建议：正文 `1.6~1.75`，紧凑信息区 `1.4~1.5`。

## 9.3 颜色与状态使用规范

- 主按钮仅使用 `accent` 或 `purple`（AI 相关动作优先 `purple`）。
- 风险/错误只使用 `danger` 语义色，不引入新红色，避免告警语义分裂。
- 状态背景优先使用 `xxxLight`，正文使用对应 `xxx`，保持“浅底深字”对比。
- `P0-P3` 与看板列颜色禁止随意改动；如需调整，必须同步更新 `priorityColor` 和 `COLUMNS`。

## 9.4 圆角、边框、阴影

- 圆角统一采用：`4(微元素) / 6(输入与按钮) / 8(卡片) / 10(下拉) / 12(弹窗)`。
- 边框统一 `1px solid C.border`，避免局部使用更深灰导致视觉噪声。
- 阴影仅用于浮层：
  - 下拉：`0 4px 20px rgba(0,0,0,0.1~0.15)`
  - 抽屉：`-12px 0 48px rgba(0,0,0,0.14~0.18)`
  - Modal：`0 20px 60px rgba(0,0,0,0.2~0.25)`

## 9.5 组件行为规范

- **Button**
  - 高度建议 `30/32/36` 三档；禁用态使用 `C.border` 背景 + `C.muted` 文本。
  - 主次按钮同组出现时，主按钮放右侧，危险操作与主按钮分离。
- **Card**
  - 默认含边框与轻阴影，Hover 增强阴影或轻微位移（不超过 `translateY(-1px)`）。
- **Tag/Badge**
  - 优先级 Tag 固定使用 `priorityColor`；计数 Badge 使用低饱和背景。
- **Drawer/Modal**
  - 打开后必须允许 ESC 或显式关闭按钮关闭；底部操作区使用 sticky，避免长内容下按钮不可见。

## 9.6 交互与动效规范

- 动效时长：`0.15s~0.3s`，默认 `ease`。
- 允许的基础动效：
  - 抽屉/弹层进入（slide/fade）
  - 列表 Hover 状态过渡
  - 进度条宽度变化
- 禁止大幅位移动画和高频闪烁，避免干扰高密度信息阅读。

## 9.7 可访问性与可用性

- 所有可点击元素必须有 `cursor: pointer` 与可见 hover/focus 反馈。
- 键盘焦点态建议统一为 `0 0 0 2px ${C.accent}55`。
- 颜色对比不足时，优先提升文字色而非加深背景。
- 状态表达不只依赖颜色，需配合文字或图标（例如“已通过/已拒绝”标签）。

## 9.8 响应式与最小可用宽度

- 工作台桌面优先，推荐最小宽度 `1200px`。
- 小屏（<=1024）建议策略：
  - 隐藏次要侧栏或切换为折叠
  - 抽屉改全屏层
  - 关键操作按钮固定在底部操作栏
- 禁止在窄屏继续保留三列并排，避免文本不可读。

## 9.9 文案与信息密度

- 操作文案使用“动词 + 对象”：如“生成 PRD”“同步 Meego”“提交 Git”。
- 状态文案保持短语化，不超过 8 个汉字。
- 高风险操作增加二次确认，并给出明确后果说明。

## 10. 参考模板版设计规范总结（强制遵循）

基于当前项目代码（主应用 + sdd/sdd2/vibe 变体）整理为可执行规范，格式参考你提供的样例。

### 设计规范（强制遵循）

#### 色彩

- 主文字：`C.ink`（浅色主题）/ `#e8ebf2`（深色主题）
- 次要文字：`C.muted` / `#93a0b4`
- 页面底色：`C.paper`
- 卡片底色：`C.white` 或 `C.cream`
- 分割线：`1px solid C.border`
- 功能色：`C.accent`（主操作）、`C.success`、`C.warn`、`C.danger`
- 业务语义色：`priorityColor(P0-P3)`、`COLUMNS[].color/bg`、`scoreColor/scoreBg`
- 禁止：同一语义引入多套颜色（例如错误态出现多种红色标准）

#### 字体

- 正文：`'Noto Sans SC', sans-serif`
- 编号/状态/技术信息：`'DM Mono', monospace`
- 标题字重：`600~800`
- 正文字重：`400~500`
- 正文行高：`1.6~1.75`
- 禁止：新增第三套主字体破坏信息层级

#### 间距与布局

- 间距基线：`4/8/12/16/24/32`
- 卡片内边距：`12~16`
- 弹窗内边距：`20~24`
- 抽屉宽度：`52~60vw`，最小 `520px`
- 页面结构：`TopNav + 主内容 + 侧栏/抽屉`
- 禁止：在窄屏继续三列并排导致正文不可读

#### 圆角与边框

- 圆角：`4/6/8/10/12`（由小到大）
- 边框：统一 `1px solid C.border`
- 卡片默认：浅边框 + 轻阴影
- 禁止：同页面出现明显不一致的圆角体系（如 2/20 混用）

#### 阴影与层级

- 下拉层：`0 4px 20px rgba(0,0,0,0.1~0.15)`
- 抽屉：`-12px 0 48px rgba(0,0,0,0.14~0.18)`
- Modal：`0 20px 60px rgba(0,0,0,0.2~0.25)`
- Toast：顶部固定，阴影中等，停留时间短
- 禁止：常规内容区大量重阴影

#### 交互

- Hover：颜色或背景轻变化，必要时 `translateY(-1px)`
- Active：轻微压感（如 `scale(0.98~0.99)`）
- Transition：`0.15s~0.3s ease`
- 焦点态：统一 `0 0 0 2px ${C.accent}55`
- 按钮分级：主（accent/purple）> 次（白底描边）> 危险（danger）
- 禁止：无反馈点击区、无禁用态样式、关键动作无二次确认

#### 动效

- 允许：抽屉滑入、弹层淡入、进度条过渡、列表 hover 过渡
- 时长：`0.15~0.3s`
- 原则：服务信息理解，不做装饰性复杂动画
- 禁止：高频闪烁、大幅位移、与业务无关循环动画

#### 可访问性

- 所有交互元素需键盘可达
- 状态表达需“颜色 + 文案/图标”双通道
- 低对比场景优先提高文字对比度
- 表单错误需明确字段级提示
- 禁止：仅靠颜色传递通过/失败/警告状态

#### 组件一致性基线

- Card：状态色条 + 标题 + 元信息 + 状态标签
- Drawer：头部操作区 + 可滚动内容区 + sticky 底部操作区
- Modal：标题/内容/操作三级结构，不省略关闭路径
- Tree：层级缩进、badge 计数、active/hover 明确区分
- Tag/Badge：用于“状态/优先级/计数”，避免承载长文本

## 12. 可直接复用的 Skills 规则块（用于 SKILL.md）

下面内容可直接复制到 Skill 文档中作为 UI 约束。

```md
# PM AI Platform UI Design Rules (Mandatory)

在本项目生成或修改前端 UI 时，必须遵循以下规则。若与现有页面冲突，优先保持现有风格一致性。

## 1) Color
- Use existing tokens from `C` only (e.g. `ink/paper/cream/white/muted/border/accent/success/warn/danger/purple/teal/sb*`).
- Primary action uses `C.accent` (or `C.purple` for AI-centric actions).
- Error/risk state must use `C.danger` family only.
- Do not introduce new semantic colors for existing states.
- Keep `COLUMNS`, `priorityColor`, `scoreColor/scoreBg` semantics unchanged unless explicitly required.

## 2) Typography
- Body/content: `'Noto Sans SC', sans-serif`.
- IDs/status/meta numbers: `'DM Mono', monospace`.
- Typical scale:
  - Page title: 20-26, weight 600-800
  - Section title: 14-16, weight 600-700
  - Body: 13-14, weight 400-500
  - Meta: 10-12
- Body line-height: 1.6-1.75.

## 3) Spacing & Layout
- Use 8pt-like spacing set: 4/8/12/16/24/32.
- Card padding: 12-16.
- Modal padding: 20-24.
- Drawer width: 52-60vw, min 520px.
- Preferred shell: top nav + main content + optional sidebar/drawer.

## 4) Radius, Border, Shadow
- Radius scale: 4/6/8/10/12.
- Border: `1px solid C.border`.
- Shadow presets:
  - Dropdown: `0 4px 20px rgba(0,0,0,0.1~0.15)`
  - Drawer: `-12px 0 48px rgba(0,0,0,0.14~0.18)`
  - Modal: `0 20px 60px rgba(0,0,0,0.2~0.25)`
- Avoid heavy shadows in normal content areas.

## 5) Interaction & Motion
- Hover: subtle color/background change; optional `translateY(-1px)`.
- Active: slight press feedback (`scale(0.98~0.99)`).
- Transition duration: 0.15s-0.3s, easing `ease`.
- Focus ring: `0 0 0 2px ${C.accent}55`.
- Motion must support comprehension, not decoration.

## 6) Accessibility
- All clickable items must provide visible hover/focus states.
- Never encode status by color alone; pair with text/icon.
- Improve text contrast first when readability is insufficient.
- Form validation must provide field-level error messages.

## 7) Component Baseline
- Card: status indicator + title + meta + status badges.
- Drawer: header actions + scrollable body + sticky bottom actions.
- Modal: title/content/actions with explicit close path.
- Tree: clear indentation + badge counts + active/hover distinction.
- Tag/Badge: for short state/count labels only.

## 8) Prohibitions
- Do not add a third primary font family.
- Do not add new color systems parallel to `C`.
- Do not use large distracting animations.
- Do not break responsive readability by forcing multi-column narrow layouts.
```

- 建议层级：页面标题 20-26（600-800），区块标题 14-16（600-700），正文 13-14，辅助 10-12。
- 正文行高建议 1.6-1.75。

## 3）间距与布局
- 统一间距档位：4/8/12/16/24/32。
- 卡片内边距 12-16；弹窗内边距 20-24。
- 抽屉宽度 52-60vw，最小 520px。
- 页面结构优先：顶部导航 + 主内容 + 侧栏/抽屉。

## 4）圆角、边框、阴影
- 圆角档位：4/6/8/10/12。
- 边框统一 `1px solid C.border`。
- 阴影建议：
  - 下拉：`0 4px 20px rgba(0,0,0,0.1~0.15)`
  - 抽屉：`-12px 0 48px rgba(0,0,0,0.14~0.18)`
  - 弹窗：`0 20px 60px rgba(0,0,0,0.2~0.25)`
- 常规内容区避免重阴影。

## 5）交互与动效
- Hover 仅做轻量变化，可选 `translateY(-1px)`。
- Active 提供轻压感（`scale(0.98~0.99)`）。
- 过渡时长 0.15s-0.3s，缓动 `ease`。
- 焦点环统一 `0 0 0 2px ${C.accent}55`。
- 动效应服务信息理解，不做装饰性炫技。

## 6）可访问性
- 可点击元素必须有可见 hover/focus 状态。
- 状态不可仅靠颜色表达，必须搭配文案或图标。
- 对比度不足时优先提升文字对比度。
- 表单错误需字段级提示。

## 7）组件基线
- Card：状态标识 + 标题 + 元信息 + 状态标签。
- Drawer：头部操作区 + 可滚动内容区 + sticky 底部操作区。
- Modal：标题/内容/操作三段结构，必须可关闭。
- Tree：层级缩进、计数 badge、active/hover 清晰。
- Tag/Badge：仅承载短文本状态与计数。

## 8）禁止项
- 禁止新增第三套主字体。
- 禁止绕过 `C` 新建平行颜色体系。
- 禁止高频闪烁或大幅位移动画。
- 禁止在窄屏强行多列导致不可读。
