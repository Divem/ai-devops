# PM AI Platform 前后端分离改造方案（Supabase）

## 1. 目标与范围

本方案用于将当前 `pm-ai-app` 的前端本地状态（`useState + localStorage`）改造为前后端分离架构，数据库使用 Supabase（Postgres + Auth + Storage + Realtime）。

目标：

- 前端：从“单文件本地状态驱动”迁移为“API/SDK 驱动”
- 后端：承接需求卡片、文档、聊天、配置、提交流程等核心业务
- 数据：统一持久化，支持权限隔离、协作、审计和扩展
- AI 与 Git：前端不再直接持有关键密钥，改由后端代理调用

---

## 2. 现状问题（基于当前代码）

当前主要状态集中在 `pm-ai-app/src/PMPlatform.jsx`：

- 需求卡片 `cards` 在内存中初始化（`INITIAL_CARDS`）
- 多类配置保存在 localStorage（`project-config`、`ai_model_*`、过滤条件等）
- AI 调用在前端完成（`callAIReview`、`callAIDoc`、`handleSendMessage`、`ai-client.js`）
- 文档与提案结构嵌套在 `card.docs` 中（`prd + proposals[]`）
- Git 提交结果写回 `gitStatus`（卡片级或 proposal 级）

这会带来：

- 无跨端同步、多人协作能力弱
- 配置和密钥存在前端暴露风险
- 业务对象耦合在大对象里，不利于查询与统计

---

## 3. 目标架构

建议采用“前端 + Supabase + BFF（Edge Functions）”三层：

1) 前端（React/Vite）
- 仅负责 UI、交互与状态缓存
- 使用 Supabase JS（查询 + 订阅）和自建 API SDK（调用 Edge Functions）

2) Supabase（数据层）
- Postgres：业务主数据
- Auth：用户/组织身份
- Storage：附件（可选）
- Realtime：卡片与文档协作更新

3) BFF（Supabase Edge Functions）
- AI 代理（持有模型 key）
- Git 提交代理（持有平台 token）
- Meego 同步代理（持有 Meego token）
- 聚合查询与权限校验

## 5. API 设计（REST + Edge Functions）

建议风格：`/api/v1/...`，前端统一通过 `apiClient` 调用。

## 5.1 需求卡片接口

- `GET /api/v1/requirements`
  - query: `projectId,status,priority,space,subsystem,app,iteration,assignee,keyword,page,pageSize`
  - resp: 列表 + 分页

- `GET /api/v1/requirements/:id`
  - resp: 卡片详情（基础信息 + 最新 ai review + 文档摘要）

- `POST /api/v1/requirements`
  - body: 标题、描述、优先级、用户故事、验收标准等

- `PATCH /api/v1/requirements/:id`
  - body: 任意可编辑字段（支持部分更新）

- `POST /api/v1/requirements/:id/move`
  - body: `{ status: "approved" }`

- `DELETE /api/v1/requirements/:id`（可选，建议软删除）

## 5.2 AI 接口（Edge Function）

- `POST /api/v1/ai/review`
  - body: `{ requirementId, model? }`
  - action: 拉取需求上下文 -> 调模型 -> 入库 `requirement_ai_reviews`

- `POST /api/v1/ai/doc-generate`
  - body: `{ requirementId, docType, proposalId?, clarificationSummary?, model? }`
  - action: 生成文档并写入 `documents`

- `POST /api/v1/ai/chat`
  - body: `{ requirementId, message, model }`
  - action: 追加聊天消息 + 返回 assistant

- `POST /api/v1/ai/clarify`
  - body: `{ requirementId }`
  - action: 返回澄清问题（可与当前 `buildReviewQuestions` 兼容）

## 5.3 文档接口

- `GET /api/v1/requirements/:id/documents`
- `POST /api/v1/requirements/:id/documents`
- `PATCH /api/v1/documents/:documentId`
- `POST /api/v1/documents/:documentId/version`
- `GET /api/v1/documents/:documentId/versions`

## 5.4 Git 集成接口（Edge Function）

- `POST /api/v1/git/commit-document`
  - body: `{ documentId, gitProfileIds?, commitMessage }`
  - 支持多仓并行提交，返回 success/failed 明细

- `GET /api/v1/git/commit-records?requirementId=...`
- `POST /api/v1/git/test-connection`

## 5.5 Meego 同步接口

- `POST /api/v1/meego/sync-one`
  - body: `{ requirementId }`

- `POST /api/v1/meego/sync-batch`
  - body: `{ projectId, from, to }`

## 5.6 配置接口

- `GET /api/v1/projects/:id/config`
- `PATCH /api/v1/projects/:id/config`
- `GET /api/v1/projects/:id/git-profiles`
- `POST /api/v1/projects/:id/git-profiles`
- `PATCH /api/v1/git-profiles/:id`

## 5.7 统计接口

- `GET /api/v1/analytics/overview?projectId=...&granularity=week|month`
  - 输出：总量、通过率、拒绝率、趋势、积压、维度分布

## 6. Supabase 实施关键点

## 6.1 RLS（行级权限）

所有业务表开启 RLS，策略基于 `organization_members`：

- 可读：项目所在组织成员可读
- 可写：member 可写需求和文档；admin/owner 可改配置
- 审计表仅 admin/owner 可见（可选）

## 6.2 实时协作

使用 Realtime 订阅：

- `requirements`：状态流转、标题更新
- `documents`：文档变更提示
- `requirement_chats`：聊天流更新

## 6.3 版本与回滚

文档编辑必须写 `document_versions`，支持：

- 任意版本回看
- 对比差异（前端 diff）
- 回滚生成新版本（不覆盖历史）

## 6.4 密钥与安全

- AI/Git/Meego token 不落前端 localStorage
- 使用 Supabase secrets + Edge Function 环境变量
- 前端只拿短期 JWT，后端按用户权限执行代理调用

## 7. 前端改造建议（分阶段）

## 阶段 A：引入基础设施

- 新增 `src/services/supabaseClient.js`
- 新增 `src/services/apiClient.js`
- 保留旧 UI，不改交互

## 阶段 B：替换卡片数据源

- `cards` 从 `INITIAL_CARDS` 改为 `GET /requirements`
- `updateCard/onMoveCard/onAdd` 改为 API 写入
- 增加加载态、错误态、空态

## 阶段 C：替换文档与聊天

- `docs` 读取改为 `documents/proposals`
- `chatHistory_*` 改为 `requirement_chats`
- 文档编辑增加版本保存

## 阶段 D：替换 AI 与 Git 调用

- `callAIReview/callAIDoc/handleSendMessage` 改调用 BFF
- Git 提交统一走 `/git/commit-document`

## 阶段 E：收尾与迁移

- 清理 localStorage 配置（保留 UI 偏好类）
- 数据回填脚本：将现有示例/历史数据导入 Supabase

## 8. 数据迁移策略

## 8.1 一次性初始化

将当前 `BASE_INITIAL_CARDS + PRD_SEEDED_SAMPLE_CARDS` 导入 `requirements`：

- `desc -> description`
- `col -> status`
- `docs.prd -> documents(doc_type=prd)`
- `docs.proposals[].{proposal,design,spec,tasks} -> proposals + documents`
- `aiResult -> requirement_ai_reviews`
- `chatHistory/chatHistory_* -> requirement_chats`

## 8.2 增量迁移

在灰度期允许“双写”：

- 前端写 API 成功后，再写 localState（仅临时）
- 验证稳定后移除本地写入

## 11. 风险与规避

- 风险：单文件组件改动面大
  - 规避：先抽 `services/hooks`，后拆 UI 子组件

- 风险：RLS 策略复杂导致误拦截
  - 规避：先最小策略 + 集成测试 + 管理员兜底查询

- 风险：AI/Git 外部调用不稳定
  - 规避：Edge Function 增加超时、重试、幂等键、失败落库

- 风险：迁移期数据不一致
  - 规避：灰度双写 + 对账脚本 + feature flag

---

## 12. 最小可落地版本（MVP）

如需快速上线，首版可只做：

- `requirements + requirement_ai_reviews + documents + requirement_chats` 四张核心业务表
- AI 仅保留 `review + chat` 两个接口
- Git 先记录提交意图，不做多仓并行
- 配置只迁移 `project` 与 `sdd`，模型密钥统一后端托管

这样可在 1.5-2 周内把“核心数据持久化 + 基础协作”先跑通，再逐步完善。
