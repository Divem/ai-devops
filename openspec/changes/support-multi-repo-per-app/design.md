## Context

当前项目仅支持 `projectConfig.git` 单仓库配置，文档提交流程一次只能写入一个仓库，无法覆盖“一个应用绑定多个仓库并行提交”的业务场景。与此同时，项目配置页已具备空间/子系统/应用关联能力（多选），可作为应用级仓库绑定的上游上下文。该变更需要在不引入后端服务前提下，完成配置模型升级、历史数据迁移、并行提交流程与结果可视化。

## Goals / Non-Goals

**Goals:**
- 支持多个 Git 仓库档案（profiles）的新增、编辑、删除与连接测试。
- 支持按应用（appId）配置多仓库绑定，并提供默认仓库兜底。
- 提交文档时按目标仓库并行写入，返回逐仓库结果并支持失败重试。
- 升级 `gitStatus` 为多仓库结构，支持部分成功展示，并保持向后兼容。

**Non-Goals:**
- 不引入服务端配置存储与权限系统。
- 不改变现有文档路径规则与仓库写入 API 协议（GitHub/GitLab）。
- 不在本次变更实现仓库级高级策略（如审批流、分支保护协商、冲突自动合并）。

## Decisions

- 决策 1：将 Git 配置升级为 profile 集合模型。
  - 方案：`projectConfig.git = { profiles, bindingsByAppId, defaultProfileIds }`。
  - 理由：可表达多仓库与复用关系，且便于应用级绑定。
  - 备选：保留单配置并追加 `secondaryRepos`。未采用，因扩展性差、难维护。

- 决策 2：提交目标仓库由“应用绑定并集 + 默认仓库兜底”解析。
  - 方案：从 `card.appIds`（兼容 `card.app`）解析目标应用，再映射到 profileId 集合并去重。
  - 理由：与项目关联模型一致，且避免“未绑定应用时无法提交”。
  - 备选：强制每个应用必须绑定仓库。未采用，因迁移期阻力较大。

- 决策 3：并行提交采用 `Promise.allSettled`。
  - 方案：逐仓库独立调用现有 GitHub/GitLab 提交函数，汇总成功/失败。
  - 理由：部分失败不阻断其余仓库，符合“多仓库并行”诉求。
  - 备选：串行提交。未采用，因耗时高且单点失败影响更大。

- 决策 4：`gitStatus` 升级为“文档 -> 仓库 -> 结果”结构。
  - 方案：`gitStatus[docType][profileId] = { ok, url, committedAt, error }`。
  - 理由：可准确表达部分成功、失败原因与重试对象。
  - 备选：仅记录成功 URL 列表。未采用，因无法保留失败上下文。

- 决策 5：采用懒迁移策略兼容旧数据。
  - 方案：加载配置时将旧 `projectConfig.git` 转换为 `legacy` profile；读取旧 `gitStatus[docType]=url` 时按默认 profile 投影。
  - 理由：用户无感升级、风险可控。
  - 备选：一次性强迁移并重写全部卡片数据。未采用，因改动范围过大。

## Risks / Trade-offs

- [风险] 多仓库并行提交可能放大 token 配置错误暴露频率。→ 缓解：提交前做目标仓库预校验并在结果中逐仓库提示。
- [风险] 配置页复杂度提升，误配概率增加。→ 缓解：拆分为“仓库档案管理 / 应用绑定 / 默认仓库”三块，并提供摘要视图。
- [风险] 旧数据与新结构共存导致渲染分支变多。→ 缓解：集中封装 normalize 函数，UI 层只消费统一结构。
- [权衡] 继续前端本地存储可快速交付，但无法跨设备共享。→ 缓解：后续可复用同结构接入后端存储。

## Migration Plan

1. 新增 `normalizeGitConfig()`，在加载配置时将旧单仓库映射到 `profiles`。
2. 新增目标仓库解析函数，优先读取 `card.appIds`，兼容 `card.app` 单值。
3. 改造 DocEditor 提交入口为多仓库并行，并输出汇总结果（成功数/失败数）。
4. 改造提案 `gitStatus` 写入与读取，兼容旧 URL 结构。
5. 配置页替换为多仓库管理与绑定 UI，并补充手工回归：旧配置恢复、多仓库并行、部分失败重试。

## Open Questions

- 当同一需求关联多个应用且绑定仓库重叠时，是否需要按应用维度保留“来源应用”元信息（当前方案仅做去重提交）。
- 是否需要在下一阶段支持“按文档类型选择目标仓库”的细粒度策略（本次不包含）。
