## Context

AI 设计页面承载了文档浏览、筛选、分组、编辑和提案操作等高频交互。当前页面存在两个交互反馈问题：一是普通区域长期显示点击手势（pointer），与真实可交互范围不一致；二是部分可操作元素缺乏稳定的悬浮/焦点视觉反馈，导致交互可发现性较差。

该变更聚焦前端交互语义与视觉反馈一致性，不改变业务流程与数据结构。

## Goals / Non-Goals

**Goals:**
- 在 AI 设计页面建立统一光标语义：默认箭头，按控件类型使用 pointer/text/resize。
- 为按钮、可点击行、树节点、筛选 chip、拖拽手柄等交互元素补齐 hover 与 focus-visible 状态。
- 保证鼠标与键盘导航都有可感知焦点反馈。

**Non-Goals:**
- 不调整文档树信息架构、筛选规则或业务逻辑。
- 不引入新的设计系统依赖或全局主题改造。
- 不变更 API、后端服务或数据存储。

## Decisions

- 将“光标语义”定义为组件级样式约束，而非页面级强制 `cursor:pointer`。
  - 理由：页面级配置容易误伤不可交互区域。
  - 备选：保留页面级 pointer 并白名单恢复箭头；未采用，维护成本高且易遗漏。

- 以“可操作元素清单”驱动 hover/focus 样式补齐。
  - 理由：可验证、可逐项回归，避免只修复局部热点。
  - 备选：仅修复用户当前反馈点位；未采用，无法避免同类问题复发。

- 焦点态优先使用 `:focus-visible` 与现有配色变量。
  - 理由：兼顾可访问性与视觉一致性，减少对鼠标用户的干扰。
  - 备选：统一使用 `:focus`；未采用，键鼠行为区分度不佳。

## Risks / Trade-offs

- [样式覆盖冲突] 某些内联样式可能覆盖新增焦点态 → 统一梳理交互元素样式优先级并做关键路径回归。
- [过度高亮造成噪音] hover/focus 效果过强可能影响阅读 → 使用现有色板的低强度背景和边框高亮。
- [遗漏交互点] 页面交互点分散，易漏改 → 以“光标类型 + 焦点反馈”双检查清单逐区验证。

## Migration Plan

1. 在 AI 设计页组件中梳理交互元素并补齐 cursor 语义。
2. 为关键交互元素添加 hover/focus-visible 样式。
3. 完成鼠标与键盘回归（默认箭头、悬浮反馈、Tab 焦点可见）。
4. 若出现视觉回归，可快速回滚到变更前样式块。
