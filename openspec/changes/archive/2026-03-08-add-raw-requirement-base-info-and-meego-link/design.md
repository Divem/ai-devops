## Context

当前看板评审抽屉已支持「原始需求 / 评审报告」双 Tab，以及原始需求正文编辑和同步到 Meego 的能力。但在实际评审场景中，参与方还需要快速确认需求归属与管理上下文（需求方、产品经理、提报时间、所属模块），并从抽屉直接跳转到 Meego 原单。

该改动主要发生在 `DetailDrawer` 的「原始需求」Tab，要求在不破坏现有编辑/同步交互的前提下增强信息可读性和可追踪性。项目当前采用本地卡片状态管理，字段可能存在缺失值，需要提供稳定的降级展示。

## Goals / Non-Goals

**Goals:**
- 在「原始需求」Tab 新增基础信息区，展示需求方、产品经理、提报时间、所属模块
- 新增 Meego 链接区，支持跳转与缺失态提示
- 兼容现有 `rawRequirement` 编辑、保存、同步到 Meego 流程
- 约束字段缺失时的统一回退展示，避免空白或报错

**Non-Goals:**
- 不改动「评审报告」Tab 内容结构
- 不新增项目级 Meego 配置页面
- 不改造后端数据源，仅在前端完成字段映射和展示

## Decisions

### 决策 1：在原始需求 Tab 中采用“三段式布局”
- 方案：基础信息区 -> Meego 链接区 -> 原始需求正文区
- 原因：先展示管理上下文，再进入正文阅读，符合评审心智顺序
- 备选：将基础信息塞入 Header
- 未选原因：Header 已承载卡片主信息，继续堆叠会降低可读性

### 决策 2：基础信息使用键值栅格展示并统一回退值
- 方案：固定四个字段标签，缺失值统一显示 `-`
- 原因：可快速横向比对，且对脏数据具备稳定容错
- 备选：缺失字段直接隐藏
- 未选原因：隐藏会造成字段认知不一致，用户无法判断是“无值”还是“无该字段”

### 决策 3：Meego 链接采用显式入口 + 空态文案
- 方案：有 `meegoUrl` 时显示可点击链接（新窗口打开）；无值时显示“未关联 Meego 链接”
- 原因：避免误点击并清晰表达当前追踪状态
- 备选：仅展示纯文本 URL
- 未选原因：可用性差，且无法明确区分可跳转与不可跳转状态

### 决策 4：字段映射优先读取结构化元数据，次选兼容旧字段
- 方案：优先读取 `card.rawRequirementMeta`（或等效结构），并兼容历史字段命名
- 原因：减少对既有数据的破坏性修改，便于逐步迁移
- 备选：一次性强制迁移全部卡片数据结构
- 未选原因：改动面过大，超出本次前端增强范围

## Risks / Trade-offs

- [Risk] 历史卡片字段分布不一致，可能出现大量缺失值 -> Mitigation：统一回退值与空态文案，并在示例数据中补齐典型字段
- [Risk] 外链地址可能非标准 URL -> Mitigation：渲染前做基础格式校验，不合法则按“未关联”处理
- [Trade-off] 兼容多种字段命名会增加前端映射逻辑复杂度 -> Mitigation：集中封装映射函数，避免散落在 JSX 中

## Migration Plan

1. 在 `DetailDrawer` 原始需求 Tab 增加基础信息与链接展示区
2. 新增/调整字段映射逻辑（结构化元数据优先，兼容旧字段）
3. 保持现有编辑与同步逻辑不变，确保回归通过
4. 更新示例数据或默认值策略，验证缺失场景展示
5. 通过手工验收覆盖“有值/缺失/非法链接”三类场景

## Open Questions

- 提报时间是否需要统一格式化（如 `YYYY-MM-DD HH:mm`）或保持原始文本？
- 基础信息字段最终命名是否固定为 `rawRequirementMeta`，还是允许继续兼容多套命名？
