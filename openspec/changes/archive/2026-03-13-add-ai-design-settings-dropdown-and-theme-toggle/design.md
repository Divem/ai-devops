## Context

当前页面存在两个问题：

1. AI 设计页与看板页顶部设置入口不一致，用户难以形成稳定操作心智。
2. 主题切换按钮与项目设置入口分离，顶部控件冗余，且“设置相关动作”没有集中收纳。

项目目前在 `PMPlatform.jsx` 内维护顶部导航、视图切换与主题状态（含 localStorage 持久化）。本次改造应尽量复用现有状态与函数，避免引入新全局状态。

## Goals / Non-Goals

**Goals:**
- 在 AI 设计页提供与看板页一致的“设置下拉入口”交互模式。
- 将“进入项目设置页”统一收敛到下拉菜单项触发。
- 将深浅色切换迁移到设置下拉中，保留现有主题切换与持久化能力。
- 保持顶部布局稳定，不影响已有模型切换与主要导航行为。

**Non-Goals:**
- 不重构 `ProjectSettingsPage` 页面内部内容与信息架构。
- 不调整主题 token、配色策略或 dark/light 渲染规则本身。
- 不引入新的配置持久化方式（继续使用既有 localStorage 逻辑）。

## Decisions

1. **统一采用“设置触发器 + 下拉菜单”入口模型**
   - 决策：AI 设计页复用看板页的设置入口形态，菜单内包含“项目设置”和“主题切换”。
   - 原因：入口一致可降低学习成本，符合“设置动作集中收纳”的交互预期。
   - 备选：保留 AI 设计页独立按钮；会继续造成跨页面不一致。

2. **项目设置改为菜单项导航，不直接暴露独立按钮**
   - 决策：仅在下拉中提供“项目设置”菜单项，点击后切换 `currentView = 'settings'`。
   - 原因：减少顶部按钮数量，明确“先开菜单再选动作”的路径。
   - 备选：保留独立“项目设置”按钮并新增菜单；会造成功能重复入口。

3. **主题切换采用菜单内二级动作（单击切换）**
   - 决策：菜单项展示当前模式（浅色/深色）并支持单击切换。
   - 原因：保留高频操作效率，同时完成入口集中；无需新增复杂弹层。
   - 备选：在菜单中再开子菜单；对当前需求过度设计。

4. **复用既有持久化逻辑与点击外部关闭行为**
   - 决策：继续使用现有 `themeMode` 与 localStorage 持久化逻辑；下拉关闭沿用已有 click-outside 模式。
   - 原因：最小改动，降低回归风险。
   - 备选：抽离全新 dropdown 状态模块；收益不高且增加重构成本。

## Risks / Trade-offs

- [风险] 下拉菜单与顶部已有下拉（如新建菜单）事件冲突 -> [缓解] 为设置下拉使用独立状态与命中区域判断。
- [风险] 主题切换迁移后用户初期不易发现 -> [缓解] 菜单项文案显式包含“主题：浅色/深色”。
- [风险] AI 设计页与看板页代码重复上升 -> [缓解] 抽取轻量复用渲染片段或共用 helper，避免复制粘贴分叉。

## Migration Plan

1. 在 AI 设计页顶部新增设置下拉触发器与菜单状态。
2. 将“进入项目设置页”逻辑挂到菜单项，移除同区块独立入口。
3. 将深浅色切换入口从页面独立按钮迁移到设置菜单项。
4. 回归验证：看板页、AI 设计页、设置页跳转、主题切换与刷新恢复。
