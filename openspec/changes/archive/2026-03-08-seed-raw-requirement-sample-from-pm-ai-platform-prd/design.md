## Context

当前系统在两个位置展示“原始需求”：
1) 看板详情抽屉的「原始需求」Tab；
2) AI 设计页面文档区的“原始需求”只读视图。

当 `card.rawRequirement` 或相关基础字段缺失时，这两个位置主要显示空态。对于演示场景，空态会降低可理解性。已有 `md/PM_AI_Plagform_PRD.md` 可作为统一示例来源，适合摘录为默认展示内容。

## Goals / Non-Goals

**Goals:**
- 在抽屉与 AI 设计页提供一致的原始需求示例兜底
- 示例内容来源于 `PM_AI_Plagform_PRD` 摘录，包含基础信息与正文
- 真实数据优先：一旦卡片有真实原始需求数据，不使用示例
- 不改变现有编辑/同步/只读规则

**Non-Goals:**
- 不实现运行时解析 Markdown 文档并动态抽取
- 不覆盖用户录入内容或 Meego 同步内容
- 不新增后端接口或持久化字段

## Decisions

### 决策 1：采用前端内置“PRD 摘录示例常量”作为兜底
- 方案：在前端定义 `RAW_REQUIREMENT_SAMPLE_FROM_PRD` 常量（基础信息 + 正文）
- 原因：稳定、可控、无运行时 IO 成本
- 备选：每次渲染实时读取 `md/PM_AI_Plagform_PRD.md`
- 未选原因：前端运行时不可直接读取仓库文件，且解析成本高

### 决策 2：统一封装“原始需求展示数据解析函数”
- 方案：增加 `getDisplayRawRequirement(card)`，输出 `isSample`、`meta`、`content`
- 原因：抽屉与设计页共享同一判定逻辑，避免展示不一致
- 备选：在两个组件各自实现 fallback
- 未选原因：逻辑分散，后续维护成本高

### 决策 3：示例展示增加轻量标识
- 方案：当命中兜底示例时显示“示例内容（来自 PRD 摘录）”
- 原因：避免用户误解为真实同步数据
- 备选：不做标识
- 未选原因：容易造成数据来源混淆

## Risks / Trade-offs

- [Risk] 示例内容过时与 PRD 不一致 -> Mitigation：将示例维护为短摘录并在变更中可更新
- [Risk] 用户误以为示例已持久化 -> Mitigation：在 UI 中显示示例标识，且不写回 card 数据
- [Trade-off] 静态摘录不如动态抽取灵活 -> Mitigation：优先保证演示稳定性，后续可升级为可配置模板

## Migration Plan

1. 定义 PRD 摘录示例常量与统一解析函数
2. 在 DetailDrawer 原始需求 Tab 接入统一解析结果
3. 在 AI 设计页“原始需求”文档视图接入同一解析结果
4. 为示例场景增加来源标识
5. 回归验证“有真实数据/无真实数据”两类展示行为
