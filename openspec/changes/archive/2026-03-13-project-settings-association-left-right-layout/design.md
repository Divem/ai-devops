## Context

当前 `ProjectSettingsPage` 的“项目配置”区块在 `pm-ai-app/src/PMPlatform.jsx` 中使用纵向表单布局依次渲染空间、子系统、应用三组多选框，最后再展示当前已选结果。该实现保留了正确的级联清理逻辑，但在多选场景下，用户无法同时观察左侧选择过程与最终配置结果，导致在较多选项时需要频繁上下滚动确认。

这个变更只调整前端信息架构，不改变 `projectConfig.project` 的存储结构，也不改变空间 -> 子系统 -> 应用的级联过滤规则。现有页面已经整体采用左右布局（左侧导航 + 右侧内容），因此区块内部继续引入稳定的左右分栏，符合当前页面风格。

## Goals / Non-Goals

**Goals:**
- 将项目关联配置区块改为更易扫描的左右结构。
- 在同一可视区域中同时呈现层级选择控件、操作引导和当前已选结果。
- 保持已有多选、级联过滤、无效项清理和保存逻辑不变。
- 为窄屏宽度提供自然退化，避免内容拥挤。

**Non-Goals:**
- 不新增新的关联层级或批量操作能力。
- 不修改 `projectConfig.project` 的字段定义与持久化方式。
- 不重构整个项目设置页面，只调整“项目配置”区块。

## Decisions

### 采用“左侧选择 + 右侧结果摘要”的两栏卡片布局
- 决策：在项目配置区块内部再拆分为两栏，左侧放置三层级选择器，右侧放置当前已选摘要、操作顺序说明和空状态提示。
- 原因：用户的主任务是完成级联选择，但确认结果同样重要；双栏布局能让选择和结果同时可见，减少来回视线切换。
- 备选方案：
  - 保持纵向堆叠，仅增强标题和分隔线：改动小，但不能解决已选结果与选择区域分离的问题。
  - 使用三列并列展示空间、子系统、应用：更接近关系浏览器，但会压缩每列宽度，不利于当前页面中的多行标签与说明文案。

### 保留现有状态与处理函数，只重组渲染结构
- 决策：继续使用 `projectSpaceIds`、`projectSubsystemIds`、`projectAppIds` 及现有 `handleToggleSpace/Subsystem/App` 逻辑，不引入新的状态模型。
- 原因：当前联动清理逻辑已经覆盖上层取消选择后的下层清理场景，变更目标是布局优化而非行为重写。
- 备选方案：
  - 抽象新的关联选择组件：复用性更高，但当前变更范围有限，会引入额外重构成本。

### 将“当前已选”升级为结构化摘要面板
- 决策：右侧使用分组摘要展示已选空间、子系统、应用，并在没有选择时给出明确占位文案。
- 原因：现有单行拼接摘要在多选项增多后可读性下降；分组展示更利于快速校验配置是否完整。
- 备选方案：
  - 保持单行文本摘要：实现最简单，但不适合左右布局下的信息密度。

### 使用响应式断点回落到纵向堆叠
- 决策：当内容区宽度不足时，两栏区域退化为单列，先展示选择器，再展示摘要。
- 原因：页面使用内联样式，最稳妥的方案是在容器样式上使用 `gridTemplateColumns`/`flexDirection` 的条件值，避免在小屏下右栏过窄。
- 备选方案：
  - 始终保持左右布局并允许横向滚动：会破坏配置表单的可读性与可操作性。

## Risks / Trade-offs

- [布局容器变复杂] -> 通过复用现有 `sectionStyle`、`multiBoxStyle` 和统一间距，避免样式碎片化。
- [右侧摘要与左侧选择信息重复] -> 将右侧内容聚焦于结果确认和操作说明，不重复渲染完整选择控件。
- [窄屏下断点不合理] -> 以当前内容区最小可读宽度为基准设计退化规则，确保在桌面窄窗下仍能完整显示。
- [只改布局但未提升层级感] -> 在左侧选择区增加分组标题、说明文案和空状态，让空间 -> 子系统 -> 应用的流程更明确。
