## Context

当前 PRD 页面“功能演示”弹框支持问题补充与校验，但提交成功后仅关闭弹框，不会把本次补充结果沉淀到 Chatbot。与此同时，真实评审补充分支已支持把澄清摘要写入 Chatbot，形成了两条路径行为不一致。

## Goals / Non-Goals

**Goals:**
- 在演示弹框提交成功后，追加一条结构化澄清摘要到当前需求的聊天历史。
- 复用现有 Q/A 摘要结构与 Chatbot 渲染样式，不新增第二套展示逻辑。
- 保持校验失败不写入、提交成功才写入的约束。

**Non-Goals:**
- 不修改弹框题目生成逻辑与字段校验规则。
- 不改动 AI 评审与提案生成流程。
- 不新增后端接口或持久化模型。

## Decisions

- 决策 1：复用已存在的 `buildClarificationSummaryPairs` 组装函数。
  - 理由：保证演示路径与真实路径摘要结构一致，降低维护成本。
  - 备选：演示路径单独拼接消息。未采用，容易格式漂移。

- 决策 2：消息类型继续使用 `clarification_summary`，仅通过 `source` 区分来源。
  - 理由：渲染逻辑保持单一分支，同时保留来源追踪能力。
  - 备选：新增 `clarification_demo_summary` 类型。未采用，增加渲染分支复杂度。

- 决策 3：沿用当前模型维度的 historyKey 写入策略（兼容 `chatHistory` 与 `chatHistory_<model>`）。
  - 理由：与现有聊天存储方式一致，避免切模型后消息缺失。
  - 备选：只写 `chatHistory`。未采用，会破坏现有多模型兼容行为。

## Risks / Trade-offs

- [风险] 演示多次提交导致聊天区信息过长。→ 缓解：每次只追加单条摘要卡片。
- [风险] 两条路径重复写入逻辑可能后续分叉。→ 缓解：统一复用摘要构建函数与消息结构。
- [权衡] 增加一次前端状态写入操作。→ 缓解：局部更新，保持最小改动。

## Migration Plan

1. 在演示弹框 `submit` 成功分支补充 chatHistory 追加逻辑。
2. 为摘要消息设置 `source: clarification_demo` 区分来源。
3. 回归验证：演示提交成功写入、校验失败不写入、多次提交可追加。
