## Context

当前看板状态流覆盖“待评审、评审中、AI 分析中、人工确认、已通过、已拒绝”，但缺少“文档已提交 Git”这一交付里程碑。实际流程中，产品在“已通过”后还会执行“提交到 Git”，没有独立状态会让协作方难以识别产品阶段是否真正收尾。

本次改动涉及看板列定义与 Git 提交成功回写逻辑，核心目标是新增 `submitted` 状态并在提交成功时自动流转，且视觉风格采用灰色系（参考待评审）。

## Goals / Non-Goals

**Goals:**
- 新增看板状态 `submitted`（文案“已提交”）并在看板中可见
- `submitted` 列使用灰色视觉风格，接近“待评审”中性语义
- Git 提交成功后自动将卡片状态更新为 `submitted`
- 保持现有状态语义兼容，不影响通过/拒绝等现有流程

**Non-Goals:**
- 不新增额外审批节点或权限控制
- 不修改 Git API 接口或提交文件路径规则
- 不改变“提交到 Git”按钮入口位置与交互文案

## Decisions

### 决策 1：新增独立状态 `submitted` 而非复用 `approved`
- 方案：看板新增 `submitted` 列
- 原因：明确区分“需求评审通过”和“文档已落 Git”两个阶段
- 备选：沿用 `approved` 并加角标
- 未选原因：信息密度高且统计/筛选语义不清晰

### 决策 2：提交成功自动流转状态
- 方案：Git 提交成功后在既有 `updateCard` 回写链路中同步 `col: "submitted"`
- 原因：状态与实际里程碑一致，减少人工二次操作
- 备选：由用户手动拖拽到“已提交”
- 未选原因：容易遗漏，影响过程可追踪性

### 决策 3：颜色与“待评审”同风格灰色
- 方案：`submitted` 使用灰色（与 `backlog` 同色系）
- 原因：表达“流程完成、低操作优先级”的中性状态
- 备选：绿色（接近成功态）
- 未选原因：会与“已通过”语义冲突

## Risks / Trade-offs

- [Risk] 部分统计视图可能未包含新状态，导致数据不一致 -> Mitigation：统一从列定义派生统计，避免硬编码状态枚举
- [Risk] Git 仅部分文档提交成功时状态判断不一致 -> Mitigation：沿用现有“提交成功”判定，仅在整体成功回调时流转 `submitted`
- [Trade-off] 新增列会增加看板横向信息量 -> Mitigation：复用现有紧凑列样式与灰色弱强调视觉

## Migration Plan

1. 在看板列配置中新增 `submitted`（文案“已提交”）
2. 设置 `submitted` 的灰色主题样式（参考 `backlog`）
3. 在 Git 提交成功处理逻辑中加入 `col: "submitted"` 回写
4. 回归验证：提交成功后卡片自动进入“已提交”列
5. 验证旧数据兼容：不含 `submitted` 的历史卡片正常渲染
