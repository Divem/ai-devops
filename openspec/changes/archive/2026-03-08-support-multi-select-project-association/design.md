## Context

项目配置页刚引入了空间、子系统、应用的关联能力，但当前数据模型和交互仍为单选，无法覆盖一个项目关联多个组织实体的场景。该变更需要在不引入后端依赖的前提下，把选择控件、状态结构、联动规则和持久化逻辑整体升级为多选。

## Goals / Non-Goals

**Goals:**
- 支持空间、子系统、应用三层配置的多选。
- 保持层级约束：上层选择变化会约束并清理下层无效已选项。
- 升级 `projectConfig.project` 数据结构并兼容旧版单选数据。
- 在项目配置区块清晰展示多选结果并维持保存成功反馈。

**Non-Goals:**
- 不新增服务端接口或远程配置持久化。
- 不变更 Git、AI、SDD、Skill 配置逻辑。
- 不引入复杂组件库，继续采用当前内联样式和原生控件方案。

## Decisions

- 决策 1：将 `projectConfig.project` 字段从单值升级为数组结构（`spaceIds` / `subsystemIds` / `appIds`）。
  - 理由：多选语义直接、序列化简单、与本地存储兼容性好。
  - 备选：保留单值字段并新增扩展字段。未采用，因会导致双轨数据和读取歧义。

- 决策 2：采用“多选 + 自动清理失效项”的层级联动策略。
  - 理由：确保下层选项始终是上层选择的子集，避免提交无效关联。
  - 备选：允许无效组合并在保存时报错。未采用，因会增加用户理解成本。

- 决策 3：兼容旧数据时将旧单值字段映射为单元素数组。
  - 理由：可平滑迁移，用户无感升级。
  - 备选：直接丢弃旧值并重选。未采用，因会造成已有配置损失。

## Risks / Trade-offs

- [风险] 原生 `<select multiple>` 在不同系统样式差异较大。→ 缓解：补充帮助文案和已选摘要，降低可用性差异影响。
- [风险] 选项量增大后，多选列表可读性下降。→ 缓解：保留层级过滤，先缩小候选集合再选择。
- [权衡] 继续本地持久化交付快，但跨设备不共享。→ 缓解：后续可在保持字段协议不变前提下接入服务端。

## Migration Plan

1. 更新配置加载逻辑：识别旧字段并转换为数组。
2. 更新项目配置区块状态与保存逻辑，确保写回新字段。
3. 通过手工验证覆盖：旧数据加载、多选保存、刷新回填、上级变更后下级清理。
4. 若异常可回滚到单选逻辑并保留转换后的数组数据，避免数据丢失。
