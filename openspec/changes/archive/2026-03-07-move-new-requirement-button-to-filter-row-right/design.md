## Context

当前看板页面的主操作按钮“新建需求”与筛选条件区域不在同一操作带，导致用户在完成筛选后发起新增时需要在页面中进行额外视线跳转。项目当前以前端单文件组件与内联样式为主（`pm-ai-app/src/PMPlatform.jsx`），适合通过局部 JSX 结构调整与样式重排完成改造。

约束如下：
- 不改变现有“新建需求”按钮点击行为、文案、权限与状态逻辑。
- 不引入新依赖，不改动后端接口或数据结构。
- 保持筛选组件交互行为不变（空间/迭代等筛选逻辑不受影响）。

## Goals / Non-Goals

**Goals:**
- 将“新建需求”按钮移动至筛选条件同一行，并固定在该行右侧。
- 在桌面端保证筛选控件与按钮同排且对齐清晰。
- 在窄屏或容器宽度不足时保持布局稳定，避免按钮遮挡或筛选控件溢出。

**Non-Goals:**
- 不调整按钮视觉风格、文案或交互反馈。
- 不改动需求创建弹窗内容与提交流程。
- 不重构看板页面其它模块（如卡片区、详情抽屉、统计区）。

## Decisions

### 决策 1：将筛选区与“新建需求”归并为同一工具行容器
- 方案：在看板页工具区中使用单一行级容器承载筛选控件与主按钮。
- 理由：最小改动即可实现用户目标，且便于维护布局一致性。
- 备选方案：仅通过绝对定位将按钮压到右侧。
- 未选原因：绝对定位对响应式和后续扩展不友好，易出现覆盖与错位。

### 决策 2：使用弹性布局实现“左筛选 + 右按钮”
- 方案：容器使用 `display: flex`，左侧筛选组可伸缩，右侧按钮组右对齐。
- 理由：与当前代码风格一致（内联样式），适配不同宽度成本低。
- 备选方案：使用 CSS Grid 严格两列布局。
- 未选原因：当前场景结构简单，Grid 增加复杂度但收益有限。

### 决策 3：窄屏下允许工具行内部换行且保持按钮在逻辑末位
- 方案：在空间不足时启用 `flex-wrap`，保证筛选优先展示，按钮保持在操作区末位。
- 理由：避免元素重叠，提升移动端可用性。
- 备选方案：窄屏隐藏按钮或移入更多菜单。
- 未选原因：会降低主操作可见性，增加点击路径。

## Risks / Trade-offs

- [风险] 不同预览入口（SDD/SDD2/Vibe）可能存在布局实现差异，导致改动不一致。  
  → 缓解：先定位主应用实现，再同步检查预览入口对应组件，确保行为一致。

- [风险] 筛选项过多时工具行拥挤。  
  → 缓解：预留换行策略与间距规则，必要时调整筛选项最小宽度。

- [取舍] 保持原有按钮行为不变意味着不借机统一工具栏交互模式。  
  → 缓解：本次聚焦布局问题，后续可独立发起工具栏体验优化变更。

## Migration Plan

1. 修改看板页工具区 JSX 结构与内联样式，完成按钮迁移。
2. 在桌面与窄屏下手工验证：筛选功能、新建需求按钮、弹窗打开流程。
3. 若发现布局回归问题，可回滚至原工具区结构（仅前端代码回退，无数据迁移成本）。
