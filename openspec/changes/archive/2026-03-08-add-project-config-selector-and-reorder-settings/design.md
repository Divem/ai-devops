## Context

当前项目配置页已经包含 AI 模型、Git、SDD 等配置能力，但缺少最上层的项目关联信息（空间、子系统、应用）。用户在执行 Git 同步或 AI 文档流程时，无法先完成统一的项目归属配置。现有默认首屏与区块顺序也未体现"先项目上下文、再集成配置"的操作路径。

## Goals / Non-Goals

**Goals:**
- 在项目配置页首位新增"项目配置"区块，用于选择空间、子系统、应用。
- 调整配置区块顺序为"项目配置 -> Git 配置 -> AI 模型配置"，降低首次配置认知成本。
- 新增项目关联配置的本地持久化与回填，保证刷新后状态一致。

**Non-Goals:**
- 不引入新的后端接口作为必需依赖。
- 不重构项目配置页整体布局与视觉风格。
- 不改动 Git 提交与 AI 请求的核心业务流程。

## Decisions

- 决策 1：项目关联配置作为 `projectConfig` 的独立字段（例如 `projectConfig.projectLinkage`）保存。
  - 理由：与 `git`、`ai`、`sdd` 一致，便于复用现有保存/加载路径。
  - 备选：单独 localStorage key 存储。未采用，因会增加状态同步成本。

- 决策 2：空间/子系统/应用采用分层选择（空间决定子系统列表，子系统决定应用列表），并提供空态兜底。
  - 理由：匹配真实数据结构，避免无效组合。
  - 备选：三个独立自由输入框。未采用，因校验困难且易产生脏数据。

- 决策 3：区块顺序通过配置页渲染顺序与默认激活项同步调整，而非仅调整文案或锚点。
  - 理由：保证导航顺序、默认内容、滚动定位行为一致。
  - 备选：只改 UI 显示顺序。未采用，因会导致导航与内容区不一致。

## Risks / Trade-offs

- [风险] 暂无后端真实数据源时，选择列表可能依赖本地样例数据。→ 缓解：先封装数据提供层，后续可无缝替换为 API。
- [风险] 旧版 `projectConfig` 无新字段时可能出现 undefined 读取。→ 缓解：在加载配置时做默认值合并与向后兼容。
- [权衡] 先做本地持久化可快速交付，但跨端一致性有限。→ 缓解：后续可在不改 UI 协议下接入服务端持久化。
