## Context

当前数据概览页仅有四个总量卡片与阶段分布条形图，属于“结果展示”，缺少“过程洞察”。产品经理无法快速回答三个关键问题：
1) 需求处理效率是变好还是变差；
2) 哪个阶段最容易堆积；
3) 哪些需求最值得优先干预。

约束条件：
- 项目为前端单文件架构，核心逻辑集中在 `PMPlatform.jsx`
- 暂无后端统计 API，需基于前端 `cards` 本地数据计算
- 需保持现有视觉风格与交互一致，不引入复杂状态管理

## Goals / Non-Goals

**Goals:**
- 提供“总览 + 趋势 + 漏斗 + 风险”四层分析能力
- 支持按优先级、负责人、空间、迭代切片分析
- 支持从统计组件一键下钻到看板并自动应用筛选
- 保持页面在中等数据量下流畅渲染

**Non-Goals:**
- 不引入后端数据仓库或离线计算任务
- 不做跨项目/跨时间段的历史快照存储
- 不实现复杂 BI 自定义报表编辑器

## Decisions

1. 数据计算策略：前端派生计算 + `useMemo`
- 选择：基于 `cards` 做聚合并缓存派生结果
- 原因：当前无后端分析 API，前端实时计算可快速落地
- 备选：新增后端统计接口（放弃，改造成本高且超出本次范围）

2. 组件结构：按“指标层级”拆分可复用块
- 选择：新增指标卡组、趋势区、漏斗区、风险区子组件（仍在同文件内）
- 原因：保持单文件风格同时降低维护复杂度
- 备选：拆分多文件（放弃，偏离当前代码组织习惯）

3. 下钻交互：写回现有筛选状态并切换到看板
- 选择：点击图表项后复用 `selectedSpaces/selectedIterations` 等筛选 state
- 原因：不新增第二套筛选系统，避免语义不一致
- 备选：新建专用 drilldown state（放弃，重复维护）

4. 图表实现：优先轻量原生实现
- 选择：先用现有样式 + div/svg 实现折线/漏斗简版
- 原因：减少第三方依赖引入风险
- 备选：引入图表库（保留为后续增强选项）

## Risks / Trade-offs

- [Risk] 前端计算在大数据量下卡顿 → Mitigation: `useMemo` 缓存 + 限制展示点位（按周聚合）
- [Risk] 本地数据缺少时间字段导致趋势失真 → Mitigation: 对缺失字段回退到 `date`，并在 UI 标注“基于当前数据估算”
- [Risk] 下钻条件覆盖用户已选筛选 → Mitigation: 提供“追加筛选/替换筛选”明确行为与 toast 提示

## Migration Plan

1. 抽取数据概览的派生指标函数（总量、趋势、漏斗、风险）
2. 增加趋势、漏斗、风险模块 UI，并接入派生数据
3. 接入下钻行为：点击统计项后切换至看板并应用筛选
4. 增加边界处理：空数据、单点数据、缺失字段提示
5. 手工回归：不同筛选组合下统计一致性与交互正确性
