## Context

AI 辅助设计页当前文档树以“产品需求 SPEC”作为起点，用户无法在同一工作区快速回看来自 Meego 的原始需求与基础信息。实际工作流中，产品需求 SPEC 是对人类原始需求的标准化转化结果，因此需要在浏览顺序上先呈现原始需求，再查看 SPEC。

## Goals / Non-Goals

**Goals:**
- 在每个需求的文档树中新增“原始需求”文档节点，并排在“产品需求 SPEC”之前。
- 点击该节点后，在编辑区域以只读形式展示原始需求基础信息与正文内容。
- 明确解释“原始需求（人类提出、Meego 同步）”与“产品需求 SPEC（标准化转化文档）”的关系。

**Non-Goals:**
- 不改动 OpenSpec 提案（Proposal/Design/Spec/Tasks）的生成与编辑逻辑。
- 不新增原始需求在线编辑能力（本次仅文档区查看）。
- 不引入新的后端接口，继续复用现有卡片字段与映射。

## Decisions

- 决策 1：将“原始需求”作为文档树顶层固定节点，位于“产品需求 SPEC”上方。
  - 原因：用户认知路径是先看输入再看转化结果，顺序更符合真实流程。
  - 备选：放在 SPEC 下方或折叠到元信息面板；未采用，因可发现性不足。

- 决策 2：原始需求文档采用只读渲染，不进入现有 Markdown 编辑器可编辑态。
  - 原因：该文档代表外部系统同步源，避免误解为可在此直接改写源需求。
  - 备选：复用编辑态并允许保存；未采用，因超出本次范围且会引入同步一致性问题。

- 决策 3：内容布局分为“基础信息区 + 原始需求正文 + 关系说明”。
  - 原因：先给结构化背景，再给完整正文，最后给语义解释，帮助理解两类文档职责。

## Risks / Trade-offs

- [Risk] 部分需求缺少 rawRequirement 或元数据字段 → Mitigation：提供空态文案与字段回退展示。
- [Risk] 用户期望直接编辑原始需求 → Mitigation：在文档内明确“来自 Meego 同步，当前仅查看”。
- [Trade-off] 增加一个顶层文档节点会略微增加树复杂度，但显著提升上下文完整性。
