## Context

当前模型配置页面允许用户填写供应商地址、模型名称和 API Key，但缺少在保存前验证配置有效性的机制。用户通常在后续真实调用模型时才暴露配置错误，导致定位成本高、失败反馈滞后。该改动聚焦于前端配置流程中的“即时可验证”能力，并复用现有请求封装与错误处理模式，避免引入新依赖或破坏既有保存流程。

## Goals / Non-Goals

**Goals:**
- 在模型配置页面提供“测试请求”操作，基于当前表单值发起一次轻量测试调用。
- 在 UI 中明确反馈测试状态（进行中/成功/失败）和可读错误原因。
- 通过按钮禁用与请求状态锁定避免重复点击造成并发测试请求。
- 保持“测试请求”与“保存配置”解耦，不改变现有保存行为。

**Non-Goals:**
- 不新增后端专用测试接口，优先复用现有模型调用能力。
- 不实现历史测试记录、监控看板或批量连通性巡检。
- 不修改模型供应商配置的数据结构与持久化协议。

## Decisions

- 新增独立的 `handleTestRequest` 前端流程，而不是复用保存动作中的校验逻辑。
  - 理由：测试与保存触发时机和失败容忍度不同，解耦可避免影响已有提交流程。
  - 备选方案：将测试强制前置为保存前校验。未采用，因会改变现有用户路径并增加阻断。
- 测试请求采用最小可行负载（如简短 prompt、低 token）调用现有模型 API。
  - 理由：可验证鉴权、模型可用性和网络连通性，同时控制耗时和费用。
  - 备选方案：构建独立 ping 接口。未采用，当前范围内实现成本更高且需后端配合。
- 引入结构化测试结果状态：`idle | testing | success | error`，并映射标准错误文案。
  - 理由：状态机简单可控，便于渲染按钮状态、提示文案及重试入口。
  - 备选方案：仅使用布尔值。未采用，难以表达中间态与错误细分。

## Risks / Trade-offs

- [测试调用产生额外 token 成本] -> 使用最小请求参数并限制返回长度。
- [部分供应商错误信息不一致] -> 在前端做统一错误分类（鉴权/模型不存在/超时/网络/未知错误）。
- [用户把“测试成功”误解为业务调用必然成功] -> 在成功提示中声明“仅验证当前配置可连通，不覆盖复杂业务场景”。
- [网络抖动导致偶发失败] -> 提供明确重试入口并保留原始错误摘要。
