## ADDED Requirements

### Requirement: 模型配置页面支持测试请求
系统 MUST 在模型配置页面提供可触发的“测试请求”能力，使用当前未保存或已修改的表单配置发起一次连通性验证调用。

#### Scenario: 使用当前表单值发起测试
- **WHEN** 用户在模型配置页面填写或修改 API 配置后点击“测试请求”
- **THEN** 系统 SHALL 使用当前表单中的配置参数发起一次测试调用，而不是读取旧的已保存配置

### Requirement: 测试请求提供明确状态反馈
系统 MUST 为测试请求展示完整状态反馈，包括进行中、成功和失败，并在失败时展示可读错误信息。

#### Scenario: 测试进行中反馈
- **WHEN** 测试请求已发出且尚未返回
- **THEN** 系统 SHALL 显示进行中状态并禁用“测试请求”按钮以避免重复触发

#### Scenario: 测试成功反馈
- **WHEN** 测试请求返回成功结果
- **THEN** 系统 SHALL 显示成功状态与成功提示文案

#### Scenario: 测试失败反馈
- **WHEN** 测试请求返回错误或请求异常
- **THEN** 系统 SHALL 显示失败状态并展示错误原因（如鉴权失败、模型不存在、网络错误、超时或未知错误）

### Requirement: 测试流程不阻断配置保存
系统 MUST 将测试请求与保存配置解耦，用户在未执行测试或测试失败的情况下仍可选择保存配置。

#### Scenario: 未测试直接保存
- **WHEN** 用户未执行测试请求直接点击保存
- **THEN** 系统 SHALL 按现有保存逻辑处理，不因测试状态缺失而阻断

#### Scenario: 测试失败后保存
- **WHEN** 用户测试请求失败后仍点击保存
- **THEN** 系统 SHALL 继续执行保存流程，并仅保留测试失败提示作为参考
