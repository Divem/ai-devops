## Why

当前模型配置页面缺少对连接可用性的即时验证，用户在填写 API 地址、模型名与密钥后无法确认配置是否可用，只能在实际生成或调用阶段才发现失败，造成反复试错与排障成本。增加“测试请求”能力可以在保存前快速验证配置正确性，降低错误配置进入主流程的风险。

## What Changes

- 在模型配置页面新增“测试请求”入口，允许用户基于当前填写的配置发起一次轻量测试调用。
- 新增测试结果反馈机制，明确展示成功/失败状态，并返回可读的错误信息（如鉴权失败、模型不存在、网络错误、超时）。
- 为测试请求增加请求中状态与防重复触发保护，避免并发点击导致重复请求。
- 在不影响现有保存逻辑的前提下，将测试能力作为独立校验流程接入。

## Capabilities

### New Capabilities
- `model-config-test-request`: 在模型配置页面提供一次性连通性测试能力，并返回结构化结果用于 UI 展示。

### Modified Capabilities
- 无

## Impact

- 受影响代码：模型配置页面相关前端组件与状态管理逻辑。
- 受影响 API：模型调用封装层需支持“测试模式”的请求与错误映射。
- 外部依赖：继续使用现有模型服务接口，无新增第三方依赖。
- 用户体验：配置阶段可提前发现错误，减少后续流程失败率。
