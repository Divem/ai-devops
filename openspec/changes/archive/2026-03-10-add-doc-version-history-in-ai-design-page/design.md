## Context

当前 AI 设计页面右侧主要包含 Chatbot 与参考资料区域，但缺少文档版本演进的可视化记录，用户很难判断某次修改是否已形成可追踪版本。虽然已有 Git 提交能力，但“提交动作”和“版本历史查看”没有在同一工作流中闭环展示。

该变更涉及页面布局扩展、版本日志数据结构落盘、以及 GitHub 提交流程与 UI 状态的联动，需要在不破坏现有文档编辑和聊天流程的前提下完成。

## Goals / Non-Goals

**Goals:**
- 在 AI 设计页右侧面板新增“版本记录”区域，并放置在 Chatbot/参考资料之后。
- 让用户可以查看当前需求文档的版本更新日志。
- 提供“提交为新版本”入口，调用现有 GitHub 提交流程并生成新版本记录。
- 在提交完成后即时回显版本信息，形成可追踪闭环。

**Non-Goals:**
- 不改造底层 Git 提交 API 协议。
- 不实现多仓库、多分支并行版本比对。
- 不引入新的后端服务或数据库。

## Decisions

1. 版本记录以“文档级版本条目列表”形式保存在当前卡片 docs 结构中。
   - Rationale: 与现有卡片数据模型一致，便于前端直接渲染和持久化。
   - Alternative considered: 单独维护全局版本中心；放弃原因是改动范围大、与当前页面上下文割裂。

2. “提交为新版本”复用现有 GitHub 提交能力，新增版本元数据写入逻辑。
   - Rationale: 降低实现风险，避免重复封装提交逻辑。
   - Alternative considered: 独立新增版本发布 API；放弃原因是当前需求仅需前端侧版本追踪。

3. 版本模块在右侧面板以第三 Tab（或顺序区域）接入，保持与 Chatbot/参考资料并列的信息架构。
   - Rationale: 用户在同一侧栏完成“讨论-参考-版本”连续动作，认知负担更低。
   - Alternative considered: 放到中间编辑区底部；放弃原因是会干扰主编辑阅读流。

## Risks / Trade-offs

- [版本记录字段设计不足，后续扩展困难] → Mitigation: 预留扩展字段（如 commitSha、docType、message、author、createdAt）。
- [提交失败但 UI 误显示成功版本] → Mitigation: 仅在 GitHub 提交成功后写入版本日志，并做错误提示。
- [右侧信息过多导致拥挤] → Mitigation: 版本列表默认折叠旧记录，优先展示最近版本。

## Migration Plan

1. 在右侧面板增加版本记录展示区域与“提交为新版本”按钮。
2. 扩展 docs 数据结构，增加 versionHistory（或等价字段）并提供兼容默认值。
3. 接入现有 GitHub 提交流程，在成功回调中写入新版本条目。
4. 回归验证：无版本数据的旧卡片可正常打开；提交成功后新版本即时可见。
5. 如需回滚，移除版本模块 UI 并忽略 versionHistory 字段读取，不影响历史卡片基础功能。

## Open Questions

- 版本号规则是否采用 `v1, v2` 递增，还是采用时间戳/语义化版本。
- 单次提交涉及多个文档（proposal/design/spec/tasks）时，版本记录按“一次提交一条”还是“每文档一条”。
