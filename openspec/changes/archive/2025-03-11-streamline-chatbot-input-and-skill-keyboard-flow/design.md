## Context

现有 Chatbot 输入区包含“Markdown 预览输入”流程，需要在预览和编辑之间切换，且 slash 技能面板键盘事件与文本输入事件耦合后，用户容易出现“想回到输入框却误触发发送”的体验问题。用户期望是更直接的对话输入：聚焦输入、按键发送、技能面板仅作为快捷选择层。

## Goals / Non-Goals

**Goals:**
- 将 Chatbot 输入区简化为单一编辑输入态，减少切换操作。
- 统一键盘行为：Enter 发送、Shift+Enter 换行。
- 在技能面板打开时支持 Tab 返回输入框且不发送。
- 在技能面板打开时 Enter 直接执行当前高亮技能并发送消息。

**Non-Goals:**
- 不调整 AI 调用模型和 prompt 内容。
- 不引入新的后端接口或状态持久化模型。
- 不重构消息渲染风格（仅保持兼容）。

## Decisions

1. **移除输入模式状态机（preview/edit）**
   - 决策：去除预览态容器和“完成编辑并返回预览”按钮，输入区固定为 textarea。
   - 原因：减少一次状态切换和认知负担，符合“聊天优先”场景。
   - 备选：保留预览但默认隐藏；仍增加状态复杂度。

2. **技能面板键盘行为分层**
   - 决策：当 slash 面板打开时，Tab 只关闭面板并将焦点留在输入框；Enter 只执行高亮技能发送；Esc 关闭面板。
   - 原因：将“选择技能”和“发送消息”语义分离，避免误发送。
   - 备选：Tab 选择技能；会与表单焦点语义冲突。

3. **发送链路保持复用**
   - 决策：技能执行和普通输入继续复用 `onSendMessage`，仅调整触发条件。
   - 原因：最小改动，避免破坏 fallback/重试/trace 现有行为。
   - 备选：独立技能发送函数；会造成双链路维护。

## Risks / Trade-offs

- [风险] 去掉预览后，用户发送前格式确认能力下降 -> [缓解] 保留 Markdown 渲染消息回显，支持 Shift+Enter 多行编辑。
- [风险] Tab 焦点管理在不同浏览器表现差异 -> [缓解] 使用显式 `preventDefault` 与 textarea ref 聚焦。
- [风险] Enter 行为变化影响习惯用户 -> [缓解] 在输入提示文案中明确按键规则。

## Migration Plan

1. 删除 Chatbot 预览输入相关 UI 和状态。
2. 调整 textarea 的 onKeyDown 逻辑，加入面板打开态下 Tab/Enter/Esc 处理。
3. 验证 slash 面板鼠标与键盘协同行为。
4. 回归普通发送、技能发送、重试、历史展示与构建。

## Open Questions

- 是否需要在后续增加“发送前格式检查”开关以替代预览模式？
- Tab 关闭面板后是否要保留当前高亮索引用于下一次打开？
