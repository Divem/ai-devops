## Context

当前平台主要使用单一视觉主题，缺少深色/浅色切换机制。随着页面模块增多（看板、AI 设计、右侧面板、弹窗等），若缺少统一主题层，后续样式维护成本会持续上升且容易出现局部不一致。

本次设计目标是在不改变业务逻辑的前提下，建立可持久化的主题切换能力，并用统一 token 驱动主要界面样式。

## Goals / Non-Goals

**Goals:**
- 提供全局深色/浅色模式切换入口并即时生效。
- 主题选择持久化（刷新后恢复）。
- 核心页面与关键交互态在两种主题下保持可读性和一致性。

**Non-Goals:**
- 不重构组件结构或业务流程。
- 不引入第三方主题库。
- 不覆盖全部历史边缘页面（优先核心主流程页面）。

## Decisions

- 使用“语义 token + 主题映射”方式组织颜色，而非在组件中写死深浅两套值。
  - 理由：可维护性更高，后续扩展主题更容易。
  - 备选：局部 if/else 切换颜色；未采用，易产生重复与遗漏。

- 主题状态提升到应用根级，并通过本地存储持久化。
  - 理由：跨页面统一、刷新可恢复，符合用户预期。
  - 备选：仅页面局部状态；未采用，切换体验割裂。

- 默认模式沿用当前视觉基线（浅色），新增深色为可选模式。
  - 理由：减少对现有用户习惯和截图文档的影响。
  - 备选：默认跟随系统；未采用，可能引起首次进入视觉突变。

## Risks / Trade-offs

- [覆盖不全] 某些内联样式未接入 token，导致深色下仍有浅色块 → 先覆盖核心路径并建立回归清单。
- [对比度不足] 深色下文字/边框可能不够清晰 → 对主要文本和交互控件做重点对比检查。
- [切换闪烁] 初始化读取本地主题前可能出现短暂闪烁 → 首屏渲染时优先读取持久化值。

## Migration Plan

1. 增加主题状态与持久化键，接入顶层页面。
2. 建立语义 token，并映射浅色/深色值。
3. 改造核心区域样式引用（看板、AI 设计、右侧面板、弹窗）。
4. 回归验证深浅模式切换、刷新恢复、交互态与可读性。
