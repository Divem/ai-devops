## Context

PRD 页面当前已支持“功能演示”入口唤起需求澄清交互，但入口放在 PRD 非空状态头部，与“先补充再生成”的用户路径不一致。用户在 PRD 空状态看到“生成提案（请先生成 PRD）”禁用提示时，正是需要引导的最佳时机。

## Goals / Non-Goals

**Goals:**
- 将“功能演示”入口迁移到 PRD 空状态区，并放在“生成提案（请先生成 PRD）”按钮下方。
- 保持点击后弹出的需求澄清交互弹框能力不变。
- 移除 PRD 非空头部中的“功能演示”入口，简化头部操作区。

**Non-Goals:**
- 不修改提案评审抽屉与提案生成主流程。
- 不改动需求澄清问题集、示例内容来源和提交校验逻辑。
- 不新增接口、配置或持久化字段。

## Decisions

- 决策 1：入口仅在 PRD 空状态可见，且位于禁用按钮下方。
  - 理由：与“先生成 PRD 再生成提案”的路径提示一致，减少错误心智。
  - 备选：保留头部与空状态双入口。未采用，易造成重复与干扰。

- 决策 2：继续复用 `ConfirmationDialog` 作为演示弹框。
  - 理由：复用现有交互与校验，避免引入新组件维护成本。
  - 备选：新建简化演示弹框。未采用，行为可能与现有澄清流程不一致。

- 决策 3：入口文案与样式保持“小字次要按钮”。
  - 理由：主路径仍是生成/更新提案，演示入口应低干扰。
  - 备选：主按钮样式。未采用，会喧宾夺主。

## Risks / Trade-offs

- [风险] 用户在 PRD 非空状态无法直接看到演示入口。→ 缓解：入口定位到空状态，契合首轮引导场景。
- [风险] 空状态区域新增元素导致视觉拥挤。→ 缓解：采用小字号、下划线文案和充足间距。
- [权衡] 迁移后发现路径更聚焦，但减少了非空状态下的即时可见性。→ 缓解：保留弹框能力，后续可按数据再评估是否补充二级入口。

## Migration Plan

1. 从 `DocEditor` 的 PRD 非空头部移除“功能演示”入口。
2. 在 PRD 空状态“生成提案（请先生成 PRD）”按钮下方新增入口并绑定现有回调。
3. 回归验证入口显示条件、弹框打开/关闭及不触发提案生成。
