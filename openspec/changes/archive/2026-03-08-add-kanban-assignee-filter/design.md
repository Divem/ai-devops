## Context

看板页面已支持空间、迭代、子系统、应用等筛选，但日常协作中经常需要按“受理人”快速查看任务负载与处理进度。当前缺少该维度，用户需要逐条卡片查找，操作成本高。

## Goals / Non-Goals

**Goals:**
- 在看板筛选条新增受理人筛选入口。
- 支持受理人与现有筛选条件叠加过滤。
- 保持筛选标签展示与“清除筛选”行为一致。

**Non-Goals:**
- 不改动卡片字段结构（沿用现有可识别的受理人字段）。
- 不引入新的后端筛选 API。
- 不调整看板列与拖拽逻辑。

## Decisions

- 决策 1：受理人候选项由当前卡片数据动态去重生成。
  - 理由：无需额外配置，随数据变化自动更新。
  - 备选：硬编码受理人列表。未采用，维护成本高。

- 决策 2：受理人筛选支持多选，并与其他筛选条件做 AND 叠加。
  - 理由：匹配现有多选筛选交互模型，便于组合查询。
  - 备选：单选。未采用，表达能力不足。

- 决策 3：清除筛选时同时重置受理人选择。
  - 理由：与现有筛选一致，避免残留状态。
  - 备选：仅重置部分筛选。未采用，行为不一致。

## Risks / Trade-offs

- [风险] 部分卡片缺失受理人字段，导致候选项不完整。→ 缓解：对缺失值不纳入候选，并保持不筛选时可见。
- [风险] 筛选维度增多导致顶栏拥挤。→ 缓解：沿用现有紧凑筛选组件样式并控制标签长度。
- [权衡] 动态候选简单易用，但无法提前筛选未来受理人。→ 缓解：后续可在有需求时补充静态候选源。

## Migration Plan

1. 增加受理人筛选状态与候选列表计算。
2. 扩展筛选组件和过滤逻辑。
3. 回归验证组合筛选、清除筛选与空结果提示。
