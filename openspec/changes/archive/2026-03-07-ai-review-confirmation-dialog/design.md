## Context

当前 AI 智能评审功能在 pm-ai-app 中通过 `callAIReview()` 函数实现。当用户点击"AI 智能评审"按钮时，系统直接调用 Claude API 进行评审，但如果原始需求信息不够详细（如缺少 userStory、acceptanceCriteria、优先级等），评审结果可能不准确或无法生成有价值的报告。

现有系统缺乏：
1. 需求信息完整性检测机制
2. 交互式信息收集 UI
3. 用户补充信息后的需求完善流程

## Goals / Non-Goals

**Goals:**
- 实现需求信息完整性检测，在评审前识别缺失的关键字段
- 设计交互式对话面板，通过选择题和填空题收集补充信息
- 支持用户填写后自动完善需求内容
- 保持与现有 AI 评审流程的无缝集成

**Non-Goals:**
- 不修改现有 AI 评审的核心算法逻辑
- 不修改飞书/Meego 数据同步逻辑
- 不涉及后端服务，所有逻辑在前端实现

## Decisions

### D1: Modal 对话框 vs 页面跳转
**选择**: Modal 对话框模式
**理由**: 用户在评审过程中不应离开当前上下文，Modal 可保持操作连续性，响应更快

### D2: 选择题优先 vs 填空题优先
**选择**: 选择题优先，填空题兜底
**理由**: 选择题降低用户输入成本，提高完成率；填空题作为补充选项处理复杂场景

### D3: 本地状态 vs 全局状态
**选择**: 组件本地状态 + useRef 持久化
**理由**: 对话面板是临时交互，不需要全局共享；使用 useRef 避免频繁重渲染

### D4: AI 问题生成策略
**选择**: 预定义问题模板 + 动态字段检测
**理由**: 预定义确保问题质量，动态检测适应不同需求场景，实现简单且可控

## Risks / Trade-offs

- **R1**: 用户可能跳过信息补充直接关闭面板 →  Mitigation: 记录未完成状态，下次评审时再次提醒
- **R2**: 补充信息后需重新提取需求内容 →  Mitigation: 复用现有需求解析逻辑
- **R3**: 选择题选项可能不覆盖所有场景 →  Mitigation: 每个选择题提供"其他"选项跳转填空题
