## Context

当前前端在 `handleAIReview` 中调用 AI 评审接口，失败后仅回退列状态并提示错误，未提供可继续演示的替代结果。由于项目需要在 API 配额不足、网络受限等场景下演示完整链路，评审失败会阻断后续“人工确认/通过”操作。

约束条件：
- 不改后端接口，不新增服务依赖
- 保持真实失败可见（仍显示错误原因）
- 兜底结果仅用于演示，必须有明确标识

## Goals / Non-Goals

**Goals:**
- AI 评审失败后提供“使用示例评审结果”入口
- 一键写入结构化示例评审数据，保持页面交互与真实结果一致
- 允许继续执行人工确认与通过流程，完成端到端演示
- 与现有错误提示、状态流转逻辑兼容

**Non-Goals:**
- 不改变 AI 评审成功路径
- 不将示例数据持久化到后端
- 不引入随机生成逻辑或新的模型调用

## Decisions

1. 兜底触发位置：评审失败后的操作区显示按钮
- 选择：在失败提示附近提供“使用示例评审结果”按钮
- 原因：用户无需离开当前上下文，操作路径最短
- 备选：自动注入示例结果（放弃，可能掩盖真实错误）

2. 示例数据结构：复用 `aiResult` 现有 schema
- 选择：严格复用 score/completeness/logic/risk/summary/risks/suggestions/passed
- 原因：复用现有渲染逻辑，不需要新增 UI 分支
- 备选：新建 `demoAiResult` 字段（放弃，增加维护成本）

3. 状态流转：示例结果写入后进入 `confirm`
- 选择：与真实评审成功一致，统一后续流程
- 原因：演示重点是全流程可走通，避免额外判定分支
- 备选：新增 `demo_confirm` 列（放弃，增加设计复杂度）

4. 演示标识：在评审报告区域显示“示例数据”标签
- 选择：显式提示“非 AI 实时生成”
- 原因：避免误导业务方将示例结果当成真实评审
- 备选：仅在 toast 提示（放弃，提示易被忽略）

## Risks / Trade-offs

- [Risk] 示例结果被误认为真实评审结论 → Mitigation: 在报告卡片内和按钮文案双重标识“示例数据”
- [Risk] 兜底能力被误用于生产决策 → Mitigation: 仅在请求失败时显示按钮，且默认文案强调“用于演示”
- [Risk] 多次点击覆盖真实结果 → Mitigation: 仅在失败状态提供入口，写入后隐藏兜底入口

## Migration Plan

1. 在 `pm-ai-app/src/PMPlatform.jsx` 增加评审示例数据常量与错误态兜底入口状态
2. 在 `handleAIReview` 失败分支记录“当前卡片可使用示例结果”
3. 新增“使用示例评审结果”动作，写入 `aiResult` 并推进到 `confirm`
4. 在评审报告 UI 增加“示例数据”提示标签
5. 手工验证：成功评审、失败后兜底、兜底后继续通过三条路径
