## Context

AI 设计页目前采用固定宽度三栏布局（左侧文档树、中间编辑区、右侧 Chatbot/参考面板）。在大屏或小屏下都存在可用性问题：固定侧栏过宽会压缩中间编辑区，固定侧栏过窄会导致导航和会话信息难以阅读。用户提出希望通过拖拽实时调整左右两侧宽度，并让中间内容自动占据剩余空间。

该变更聚焦前端布局层，不涉及后端接口、数据模型与 AI 调用协议。实现位置主要在 `pm-ai-app/src/PMPlatform.jsx` 的 `DetailPage` 主布局。

## Goals / Non-Goals

**Goals:**
- 左侧导航栏与右侧面板支持鼠标拖拽改宽。
- 中间 DocEditor 区域在拖拽过程中实时自适应剩余空间。
- 侧栏宽度存在最小/最大边界，避免布局不可用。
- 用户调整后的宽度支持本地持久化，刷新或重新进入后可恢复。
- 在移动端或窄屏下保证页面不崩坏（可降级为固定或限制拖拽）。

**Non-Goals:**
- 不实现复杂分栏库替换（如引入第三方 split-pane 依赖）。
- 不改变左右栏内部业务组件行为（DocTree、Chatbot、Reference 的业务逻辑不变）。
- 不引入服务端同步用户布局偏好。

## Decisions

### D1. 使用原生 pointer/mouse 事件实现拖拽
- 决策：在左右栏边界增加拖拽 handle，监听 `mousedown/mousemove/mouseup`（或 pointer 等价）更新宽度 state。
- 原因：现有项目为单文件 React 风格，原生实现最轻量，便于在当前架构中快速集成。
- 备选：引入 split-pane 三方库；被拒绝，增加依赖和样式适配成本。

### D2. 宽度采用受限区间并按像素持久化
- 决策：定义 `leftWidth/rightWidth` 的 min/max（如左 220-420，右 280-520），通过 `clamp` 约束；存入 localStorage。
- 原因：像素值与现有内联样式体系一致，易于在桌面端维持稳定体验。
- 备选：百分比存储；在复杂窗口尺寸变化下可预测性较差。

### D3. 中间区使用 flex 自适应
- 决策：左右两栏固定当前像素宽度，中间编辑区 `flex: 1` 自动填充剩余空间。
- 原因：改动最小，不影响现有子组件通信与渲染逻辑。

### D4. 窄屏降级策略
- 决策：当容器宽度小于阈值时禁用拖拽并回退到安全宽度组合（或仅允许单侧收缩）。
- 原因：避免在小屏设备上出现中间区宽度过小导致不可编辑。

## Risks / Trade-offs

- [拖拽事件泄漏] → 在 `mouseup`/组件卸载时统一解绑事件。
- [宽度状态与窗口 resize 冲突] → 在窗口尺寸变化时重新 clamp 当前宽度。
- [过多重渲染] → 仅更新必要宽度 state，避免在拖拽中触发无关状态计算。
- [可访问性不足] → 为拖拽条增加可视反馈与 hover/cursor 提示，后续可补键盘调整。

## Migration Plan

1. 在 `DetailPage` 增加左右宽度 state、默认值和 localStorage 读写。
2. 在左右栏与中间区之间插入拖拽 handle 并接入事件处理。
3. 应用 `clamp` 与窗口 resize 自适应策略。
4. 完成手工验证：拖拽、刷新恢复、窄屏降级、各面板功能回归。
5. 若出现问题，回滚到固定宽度配置（移除 handle 和宽度 state 即可）。
