## Context

**当前状态**: `DocTreeSidebar` 组件直接遍历 `DOC_TYPES` 数组，平铺显示所有文档类型。数据结构为 `docs: { prd, spec, proposal, design, tasks }`。

**问题**: 无法支持多提案场景，一个 PRD 拆解成多个技术提案时，用户无法区分哪些文档属于哪个提案。

**约束**:
- 保持现有单提案场景的 UI 体验
- 不能破坏现有卡片的显示
- 代码在一个 JSX 文件中，没有外部依赖

## Goals / Non-Goals

**Goals:**
- 支持多提案文件夹结构显示
- 每个提案文件夹可展开/收起
- 保持向后兼容（旧数据结构仍能正常显示）
- 单提案场景自动降级为平铺显示

**Non-Goals:**
- 不涉及后端 API 变更
- 不修改文档编辑/预览逻辑
- 不改变文档类型的定义（DOC_TYPES）

## Decisions

### 1. 数据结构设计

**决策**: 使用 `docs.proposals` 数组替代平铺结构

**理由**:
- 数组天然支持多个提案
- 每个提案独立存储，便于管理和扩展
- 保持与 `openspec/changes/<name>/` 的映射关系

**新旧结构对比**:
```javascript
// 旧结构（向后兼容）
docs: { prd: "...", spec: "...", proposal: "...", design: "...", tasks: "..." }

// 新结构（多提案）
docs: {
  prd: "...",
  proposals: [
    { id: "auth", name: "认证系统", proposal: "...", design: "...", spec: "...", tasks: "..." },
    { id: "payment", name: "支付模块", proposal: "...", design: "...", spec: "...", tasks: "..." }
  ]
}
```

### 2. UI 渲染策略

**决策**: 根据数据结构自动选择渲染模式

**渲染规则**:
1. 如果 `docs.proposals` 存在且长度 > 0 → 使用文件夹模式
2. 否则 → 使用平铺模式（向后兼容）

**文件夹模式结构**:
```
📋 产品需求 SPEC
📁 OpenSpec 提案
  ├── 📁 提案1/
  │   ├── 💡 Proposal
  │   ├── 🏗 Design
  │   ├── 📝 Delta Spec
  │   └── ✅ Tasks
  └── 📁 提案2/
      └── ...
```

### 3. 状态管理

**决策**: 使用 `expandedProposals` Set 跟踪展开状态

**理由**:
- Set 数据结构提供 O(1) 查找性能
- 轻量级状态，不需要持久化
- 展开/收起操作简单高效

### 4. 图标选择

**决策**: 使用文件夹图标 📁 表示提案组

**理由**:
- 符合用户对文件夹的认知模型
- 与现有文档类型图标区分明显
- 折叠时显示 📁，展开时显示 📂

## Risks / Trade-offs

| 风险 | 缓解措施 |
|------|----------|
| 数据迁移复杂 | 提供迁移函数 `normalizeDocs()` 自动转换旧数据 |
| 用户不理解新结构 | 单提案场景自动降级，保持原有体验 |
| 渲染性能问题 | 使用 React memo 优化，避免不必要的重渲染 |
| 展开状态丢失 | 使用 useState 管理展开状态，组件级别存储 |

## Migration Plan

1. **数据模型迁移**
   - 创建 `normalizeDocs()` 函数检测并转换旧数据结构
   - 在卡片加载时自动调用

2. **UI 组件更新**
   - 修改 `DocTreeSidebar` 添加提案文件夹渲染逻辑
   - 添加展开/收起交互

3. **示例数据更新**
   - 创建一个包含多提案的示例卡片
   - 验证 UI 显示效果

4. **验证测试**
   - 测试旧数据卡片仍能正常显示
   - 测试新数据卡片的文件夹交互
