## Context

当前项目配置页中的 AI Skill 区块仅支持按 skillId 编辑一段 prompt 文本，数据结构是 `projectConfig.skills[skillId] = string`。这与 Anthropic Skills 常见的目录化资产形态（`SKILL.md` + 参考文档 + 脚本）不一致，导致示例复用、结构校验和后续导出能力都较弱。该变更需要在保留现有可用性的前提下，把 Skill 管理升级为结构化文件视图，并内置项目管理场景示例内容。

## Goals / Non-Goals

**Goals:**
- 在 AI Skill 页面提供标准化结构：`SKILL.md`、`references/*`、`scripts/*`。
- 提供可一键填充的示例文件内容（需求澄清、Git 同步检查、字段映射、示例脚本）。
- 升级本地数据结构以保存多文件内容，并兼容旧版纯字符串配置。
- 增加路径规范校验，自动纠正常见拼写错误（`scrips` -> `scripts`）。

**Non-Goals:**
- 不接入真实文件系统写入，不直接在仓库创建物理 skill 文件。
- 不引入后端持久化，继续使用 localStorage。
- 不改造 AI 调用引擎协议，仅调整 Skill 配置来源与组装方式。

## Decisions

- 决策 1：Skill 配置模型从字符串升级为结构化对象。
  - 方案：`projectConfig.skills[skillId] = { files: { "SKILL.md": string, "references/*.md": string, "scripts/*.ts": string } }`。
  - 理由：最小改动支持多文件，且便于渲染文件树。
  - 备选：把每个文件拆成独立顶层键。未采用，因命名复杂且扩展性差。

- 决策 2：UI 采用“左侧 skill 列表 + 中间文件树 + 右侧编辑器”三栏。
  - 理由：与开发者对文件结构的心智模型一致，便于快速切换。
  - 备选：单下拉切换文件。未采用，因可见性和操作效率较差。

- 决策 3：内置示例模板按“初始化填充 + 可重置”提供。
  - 理由：保证首次进入即有可编辑内容，并可恢复官方推荐结构。
  - 备选：仅空模板。未采用，因用户理解成本更高。

- 决策 4：旧数据采用懒迁移。
  - 方案：当读取到旧字符串时，自动封装到 `SKILL.md` 文件正文并打上兼容标记。
  - 理由：避免已有自定义 prompt 丢失。
  - 备选：一次性迁移全部键。未采用，因边界与回滚复杂。

## Risks / Trade-offs

- [风险] 三栏布局在小屏幕上可读性下降。→ 缓解：窄屏降级为两步切换（文件树抽屉化）。
- [风险] 结构化对象增加状态复杂度。→ 缓解：统一封装 normalize/serialize 函数，避免 UI 直连原始结构。
- [风险] 用户可能输入不规范路径。→ 缓解：保存前执行路径白名单校验并自动修正常见拼写。
- [权衡] 继续 localStorage 快速落地，但跨设备不共享。→ 缓解：后续可在模型不变前提下接入后端。

## Migration Plan

1. 增加 `normalizeSkillConfig`，把旧字符串配置映射到 `files["SKILL.md"]`。
2. 初始化示例模板与路径校验规则，确保默认目录结构一致。
3. 改造 AI Skill 页面为文件树编辑模式并接入保存/恢复默认。
4. 将 AI 调用读取逻辑更新为优先读取 `SKILL.md` 正文（兼容旧值）。
5. 回归验证：旧配置回填、示例内容生成、目录拼写修正、刷新持久化。
