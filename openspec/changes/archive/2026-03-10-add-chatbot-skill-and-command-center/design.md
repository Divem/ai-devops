## Context

当前 Chatbot 已具备基础对话、Markdown 渲染、失败兜底和思考轨迹能力，但缺少“面向产品经理场景”的快捷能力入口。用户需要记忆提示词、手动组织分析维度，使用门槛偏高，且不同人输出格式不一致。

本次设计目标是在不破坏现有聊天链路的前提下，增加 slash 触发的技能/命令中心，让输入、执行、展示形成稳定闭环。

## Goals / Non-Goals

**Goals:**
- 在 Chatbot 输入框中输入 `/` 时弹出技能/命令选择面板。
- 内置产品经理常用技能：需求评审、风险检查、相关需求、总结功能点、完备性检查。
- 统一技能与命令的数据结构与执行入口，支持后续扩展。
- 执行结果以结构化消息渲染，保留追溯信息（来源、耗时、状态）。

**Non-Goals:**
- 不实现远端技能市场或在线安装。
- 不做复杂权限体系与多角色隔离。
- 不在本期引入后端持久化，先沿用前端本地配置与会话状态。

## Decisions

1. **统一注册表模型（Skill + Command）**
   - 决策：在前端维护 `CHATBOT_TOOL_REGISTRY`，条目包含 `id/type/title/description/promptTemplate/icon/tags`。
   - 原因：便于面板统一渲染、统一搜索过滤、统一执行分发。
   - 备选：分散在多个常量与分支逻辑中；缺点是扩展成本高、配置不集中。

2. **Slash 面板采用输入态内联浮层**
   - 决策：当输入框内容匹配 `/` 前缀时显示浮层，支持上下键选择与 Enter 确认。
   - 原因：符合用户在截图中期望的命令面板交互，学习成本低。
   - 备选：独立按钮打开弹窗；缺点是操作路径长，且和“命令输入心智”不一致。

3. **执行链路与普通聊天复用**
   - 决策：选中技能/命令后构建结构化用户消息，再走现有 `handleSendMessage` 主链路；新增 `source=skill|command` 元信息。
   - 原因：可复用已有 loading/fallback/thinking trace 机制，降低改造风险。
   - 备选：单独开一套执行函数；缺点是状态分叉，后续维护复杂。

4. **默认 PM 技能内置 + 可被配置覆盖**
   - 决策：内置默认能力作为兜底，同时允许从 AI Skill 配置中读取同名覆盖项。
   - 原因：开箱即用，同时保留团队定制空间。
   - 备选：仅支持自定义配置；缺点是首次使用不可用。

## Risks / Trade-offs

- [风险] 面板交互引入后可能与 Enter/Shift+Enter 冲突 -> [缓解] 约定面板激活时 Enter 优先用于选择，发送仅在面板关闭时触发。
- [风险] 技能 prompt 过长影响响应时延 -> [缓解] 预置模板控制长度，并在执行前做上下文裁剪。
- [风险] 新消息类型导致旧会话渲染异常 -> [缓解] 保持向后兼容，未知类型回退纯文本渲染。
- [风险] 默认技能结果质量波动 -> [缓解] 为每个技能固定输出结构，必要时使用示例兜底。

## Migration Plan

1. 新增注册表与默认技能定义，不影响旧消息结构。
2. 接入输入框 slash 检测与面板渲染，默认关闭状态。
3. 将技能/命令执行接入现有发送链路，补充消息元信息。
4. 增加结构化结果渲染与失败兜底验证。
5. 回归普通聊天、Markdown 预览、模型切换、历史会话兼容性。
