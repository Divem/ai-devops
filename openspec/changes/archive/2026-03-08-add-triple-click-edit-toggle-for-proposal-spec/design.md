## Context

当前主流程文档预览与编辑位于 `DetailPage -> DocEditor`，用户查看 Proposal/Spec 后需移动到顶部按钮点击“✎ 编辑”才能进入编辑态。该路径在高频审阅场景下存在操作中断。项目为 React 单文件组件和内联样式模式，适合以局部交互增强方式落地，不涉及数据模型或 API 变更。

约束：
- 仅增强交互，不改变保存、取消、写回逻辑。
- 仅针对 Proposal 与 Spec 文档类型，不影响 PRD/Design/Tasks。
- 不引入新依赖。

## Goals / Non-Goals

**Goals:**
- 在 Proposal/Spec 预览态支持三连击快速进入编辑态。
- 保持现有“✎ 编辑”按钮入口可用。
- 避免误触发和跨文档类型副作用。

**Non-Goals:**
- 不重构编辑器或 Markdown 渲染器。
- 不改动 DesignStudio 旧路径（本次只覆盖主路径 DocEditor）。
- 不新增手势配置面板或个性化阈值设置。

## Decisions

### 决策 1：在 `DocEditor` 预览容器层做三连击检测
- 方案：在预览态容器上记录点击时间窗内的点击次数，达到 3 次时触发 `onEditMode`。
- 原因：改动最小，复用现有编辑切换逻辑。
- 备选：在全局监听 `document` 点击。
- 未选原因：作用域过大，容易误触。

### 决策 2：仅 `docType in {proposal, spec}` 生效
- 方案：在触发前做文档类型白名单判断。
- 原因：满足业务目标并避免影响其它文档。
- 备选：所有文档都支持三连击。
- 未选原因：需求未要求，且可能引入意外行为变化。

### 决策 3：编辑态禁用三连击监听
- 方案：仅在 `!editMode && content` 时启用三连击行为。
- 原因：避免用户在编辑过程中被重复触发。
- 备选：编辑态仍监听但不执行。
- 未选原因：增加无效事件处理，无收益。

## Risks / Trade-offs

- [风险] 用户在阅读中连续点选可能误进入编辑。  
  → 缓解：设置较短时间窗，仅在 Proposal/Spec 生效。

- [风险] 与文本选择动作冲突。  
  → 缓解：仅绑定到渲染容器点击，命中后才切换，不改变文本渲染结构。

- [取舍] 不覆盖旧 DesignStudio 路径会导致两套体验不一致。  
  → 缓解：先覆盖主路径，若旧路径恢复使用再单独补齐。

## Migration Plan

1. 在 `DocEditor` 添加三连击状态与触发逻辑。
2. 手工验证 Proposal/Spec 三连击可进入编辑态。
3. 验证其它文档类型、按钮编辑入口与保存流程无回归。
4. 如有问题可直接回退该交互逻辑，不涉及数据迁移。
