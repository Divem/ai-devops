## Context

当前 PRD 文档页已提供“生成提案/更新提案”主操作，但用户在生成前缺少就地的澄清引导入口。系统已有需求澄清交互能力，可复用为“功能演示”弹框，减少用户在生成前的认知负担，并提升高质量输入的完成率。

## Goals / Non-Goals

**Goals:**
- 在 PRD 文档页主按钮下方新增低干扰“功能演示”小字入口。
- 点击入口后打开需求澄清交互式弹框，复用现有澄清问题流程。
- 保持现有“生成提案/更新提案”与评审抽屉流程不变。

**Non-Goals:**
- 不改动 callAIReview/callAIDoc 的调用顺序与容错逻辑。
- 不新增后端接口或持久化字段。
- 不重构需求澄清问题集与评分策略。

## Decisions

- 决策 1：入口位置放在“生成提案/更新提案”按钮下方，采用小号次要文案按钮。
  - 理由：与主流程强关联，且视觉层级低于主按钮，不打断主任务。
  - 备选：放到页面右侧帮助区。未采用，距离主操作远、发现路径长。

- 决策 2：弹框复用现有需求澄清交互组件与状态机，仅新增触发来源。
  - 理由：复用成本低，交互一致性高。
  - 备选：新建独立演示弹框。未采用，会引入重复维护。

- 决策 3：弹框关闭后停留在当前 PRD 文档，不自动触发提案生成。
  - 理由：避免误触发高成本生成动作，保持用户可控。
  - 备选：关闭后自动生成。未采用，风险高且不可预测。

## Risks / Trade-offs

- [风险] 用户误把“功能演示”当作主操作。→ 缓解：弱化样式、文案明确为演示。
- [风险] 弹框与既有评审抽屉状态冲突。→ 缓解：使用独立开关，避免共享 loading 状态。
- [权衡] 增加一个入口提升引导，但头部区域信息密度上升。→ 缓解：保持小字、紧邻主按钮且不占主操作行。

## Migration Plan

1. 在 `DocEditor` 的 PRD 操作区下方新增“功能演示”入口。
2. 在 `DetailPage` 增加弹框开关与触发回调，并挂载澄清交互式弹框。
3. 回归验证：入口显示条件、点击打开/关闭、与“生成提案”流程互不干扰。
