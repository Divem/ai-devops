## Context

目前“原始需求”在详情/设计区域以普通文本显示，无法体现 Markdown 结构，导致可读性与信息层次不足。同时在部分文档点击编辑后，编辑区初始化文本为空，用户误以为内容丢失。该变更聚焦展示与编辑初始化两个稳定性问题，不改业务流程。

## Goals / Non-Goals

**Goals:**
- 让原始需求正文按 Markdown 语义渲染，提升阅读体验。
- 修复点击编辑后文本被清空问题，确保编辑态沿用当前文档内容。
- 保持“原始需求”只读约束，不引入可编辑行为。

**Non-Goals:**
- 不变更原始需求来源或同步机制。
- 不重构 Markdown 渲染引擎或引入新第三方库。
- 不调整文档树结构与导航逻辑。

## Decisions

- 决策 1：原始需求正文渲染复用现有 MarkdownRenderer，而非新建轻量渲染器。
  - 理由：项目已有稳定渲染路径，统一表现且实现成本低。
  - 备选：仅保留换行文本。未采用，因层级和可读性不足。

- 决策 2：编辑模式入口统一使用“当前选中文档内容”初始化编辑态文本。
  - 理由：保证编辑前后内容一致，避免空值覆盖。
  - 备选：从 docs 字段重新拉取。未采用，因易与当前视图状态不一致。

- 决策 3：对空内容做显式保护，只有内容确实存在时才进入编辑写入流程。
  - 理由：减少边界态导致的空文本闪断。
  - 备选：不做保护直接进入编辑。未采用，因会放大空值风险。

## Risks / Trade-offs

- [风险] 原始需求中存在非标准 Markdown 语法，显示可能与预期不一致。→ 缓解：保留原文兜底与空态提示。
- [风险] 编辑初始化逻辑改动可能影响其他文档类型。→ 缓解：回归验证 prd/spec/proposal/design/tasks 全链路。
- [权衡] 复用现有渲染器可快速交付，但样式细粒度定制有限。→ 缓解：先保证语义渲染正确，样式后续迭代。

## Migration Plan

1. 调整原始需求内容展示路径，接入 Markdown 渲染。
2. 修复编辑态初始化逻辑并添加空值保护。
3. 回归验证原始需求展示、编辑进入/取消/保存流程。
