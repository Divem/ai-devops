## Context

AI 辅助设计页面左侧文档树属于暗色侧栏区域，需与右侧浅色编辑区形成稳定对比。当前存在亮色底泄漏到侧栏导航的问题，导致文档树层级与选中态辨识下降，且与既有暗色视觉语言不一致。

此次改动是样式一致性修复，目标是恢复暗色侧栏主题并约束交互态在暗底下的可读性。

## Goals / Non-Goals

**Goals:**
- 恢复文档树容器与节点在暗色主题下的背景一致性。
- 明确并统一 default / hover / active / focus-visible 四种状态的颜色规则。
- 保证筛选 chip、分组下拉、树节点文本与图标在暗色背景下有足够对比度。

**Non-Goals:**
- 不变更文档树结构、分组逻辑和筛选逻辑。
- 不重构全局主题系统或引入新设计 token。
- 不涉及 API、状态管理和数据模型调整。

## Decisions

- 以“侧栏背景色源头”统一修复，而非零散补丁。
  - 理由：亮底问题通常来自局部样式覆盖，统一入口更稳定。
  - 备选：仅修复当前可见节点样式；未采用，后续新增节点仍可能回归。

- 将交互态色彩约束绑定现有暗色变量（如 `C.sb`/`C.sbHover`/`C.sbActive`/`C.sbText`）。
  - 理由：与现有页面风格一致，降低视觉断层。
  - 备选：引入新色值；未采用，增加维护成本并易与既有主题冲突。

- 焦点态使用高可见轮廓但不改变布局。
  - 理由：保证键盘可访问性且避免元素跳动。
  - 备选：使用边框替换；未采用，可能引发布局抖动。

## Risks / Trade-offs

- [局部覆盖未清理干净] 某些内联样式仍可能写死浅色背景 → 逐层检查容器、节点、控件并统一来源。
- [对比不足] 暗色文本或标签在某些状态下仍不清晰 → 通过最小对比阈值回归（重点检查 hover/active）。
- [影响其他侧栏模块] 共享类名可能波及右侧面板 → 将样式作用域限制在 AI 设计页左侧树区域。

## Migration Plan

1. 修正文档树容器与节点背景来源，移除亮色覆盖。
2. 统一交互态颜色与焦点态样式。
3. 回归验证筛选、分组、展开、选择等主流程视觉一致性。
4. 若发现非预期影响，按组件范围回滚样式变更。
