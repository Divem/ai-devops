## Context

当前 `ChatbotPanel` 使用单行输入框并以纯文本方式展示大部分会话内容。对于包含 Markdown 语法的输入（如标题、列表、代码块），用户只能发送后再观察结果，缺少发送前确认机制。需求希望在不改变 AI 调用协议的前提下，引入预览模式并提升消息渲染可读性。

项目内已存在 `MarkdownRenderer`，可用于文档区渲染。本次改动主要集中在 `pm-ai-app/src/PMPlatform.jsx` 内的 Chatbot 输入区与消息区。

## Goals / Non-Goals

**Goals:**
- 在 Chatbot 输入区提供“编辑/预览”双模式切换。
- 预览模式下实时展示当前输入的 Markdown 渲染效果。
- assistant 消息支持 Markdown 渲染，保持文本可读性与结构化展示。
- 对空输入、异常语法和超长内容提供稳定降级策略。

**Non-Goals:**
- 不引入新的第三方编辑器（如 Monaco/TipTap）。
- 不变更消息存储结构与后端接口协议。
- 不实现复杂富文本编辑功能（仅做 Markdown 输入预览）。

## Decisions

### D1. 输入框升级为多行 textarea + 预览切换
- 决策：将当前 Chatbot 输入从 `input` 升级为 `textarea`，并新增 `edit/preview` 切换状态。
- 原因：Markdown 内容通常多行，textarea 才能满足可编辑性；切换模式便于发送前确认。
- 备选：保留单行输入，仅发送后渲染；被拒绝，无法满足“预览模式”核心诉求。

### D2. 复用现有 MarkdownRenderer 渲染输入预览与 assistant 消息
- 决策：预览区域和 assistant 气泡统一使用 `MarkdownRenderer`，降低样式和语法差异。
- 原因：项目已有渲染组件，复用可减少重复逻辑与维护成本。
- 备选：手写轻量 Markdown 解析；被拒绝，易出现语法覆盖不一致。

### D3. 增加渲染容错与性能保护
- 决策：对空文本显示占位提示；对异常渲染场景回退为纯文本显示；对超长内容可裁剪预览或显示提醒。
- 原因：避免预览模式导致界面卡顿或不可用。

### D4. 发送行为与模式约定
- 决策：发送后清空输入并自动切回编辑模式，减少连续发送时误操作。
- 原因：符合当前会话型交互习惯，且行为明确可预测。

## Risks / Trade-offs

- [Markdown 渲染与现有消息样式冲突] → 对消息容器增加统一排版约束（间距/换行/代码块滚动）。
- [长文本预览性能下降] → 设置安全阈值并在超阈值时提示“建议直接发送查看完整渲染”。
- [快捷键行为变化] → 明确 Enter/Shift+Enter 规则并在输入区提示。

## Migration Plan

1. 在 ChatbotPanel 增加输入模式 state 与切换 UI。
2. 将输入控件升级为 textarea，并接入预览区域。
3. assistant 消息切换为 Markdown 渲染并保留容错回退。
4. 调整发送后的输入状态与模式重置逻辑。
5. 完成手工回归（纯文本、Markdown、多行、超长、异常语法）。

## Open Questions

- 用户消息是否也默认按 Markdown 渲染？本次建议先 assistant 渲染，用户消息保持气泡纯文本可读。
- 是否需要记住用户上次“编辑/预览”模式偏好？本次默认不持久化。
